7.6.     ISR寫作注意事項

 

菜鳥：『聽你說了這麼多，我覺得只要觀念對的話，驅動程式的寫作好像並不是那麼困難。ˊˊ用台　』

 

韌體老鳥：『不是我老王賣瓜，驅動程式開發還是有比較高的門檻，世上電子設備與IC何止萬千，對韌體工程師而言，某個產品裡用到的CPU或IC，此後可能就不會再碰到了，所以除了8051、ARM…這些在業界已有廣泛應用的CPU外，其他CPU真的不容易找到參考書，更不用說非CPU的IC了，工程師可依賴的就只有原廠的data sheet與sample code而已；雖說觀念是一樣的，但還是有一些靠經驗累積來的小技巧。驅動程式開發有兩個地方和一般程式比較不一樣的地方，一個是中斷處理程式(ISR)，另一個是測試手法。ˊˊ用台　』

 

菜鳥：『ISR的寫作是比較麻煩，但除了要準備中斷向量表外，所謂的ISR不就也是個函數嗎，還有什麼該注意的？ˊˊ用台　』

 

韌體老鳥：『寫作不嚴謹的ISR很容易造成系統不穩定，因為硬體事件可能在任何時間點發生，亦即任何程式段在執行時都可能被ISR中斷，所以不見得可以重現這個問題，所以很難除錯。

 

此外有一個重要的概念是：CPU執行程式的基本單位是組合語言碼，而不是C語言程式段，例如一行 “a = b + 5 ;”的C語言指令可能對應到好幾行組合語言指令，在這些組合語言指令之間都可能產生中斷。我舉個極端的例子好了：

 

B = 10 ;

A = B + 5 ;

 

假設第二行C語言程式對應到以下幾行CPU執行的基本指令：

 

1.          將位址“B”內的值置於暫存器R0

2.          將R0的內容加5

3.          將R0的值存到位址“A”中

 

在1之前、1與2之間、2與3之間以及3之後都可能會產生中斷，你想想如果在這些點上產生的中斷，而且ISR動到了全域變數A或B會發生什麼事情？ˊˊ用台　』

 

菜鳥：『如果在步驟1之前產生中斷，而且ISR把“B”設為12的話，那麼上述程式執行結果會變成12+5等於17，而不是預料中的10+5等於15。系統表現出來的現象就是所謂的“不穩定”，大部份時間執行結果都是正確的，偶而會出現幾次錯誤；我可以理解你說這種問題很難除錯的原因了，如果我們一行一行的trace程式，只要中斷不“適時”產生，則錯誤的狀況根本不可能重現，根本無從trace起。ˊˊ用台　』

 

韌體老鳥：『沒錯，所以ISR寫作注意事項第一條：盡量不要在ISR內使用全域變數，如果一定要使用的話一定要做好保護，我們稱這種應該受保護的程式區段為critical section。保護的方法很多，最簡單的是在進入critical section之前，先禁止中斷的產生，所以在critical section中全域變數就不會被ISR破壞，從而達到保護的效果。請見下例：ˊˊ用台　』

 

 

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

// Function: 一般函數

//     利用禁止中斷來保護critical section

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

void fun(void)

{

 ...

 // 禁止中斷產生

 //

 disable_interrupt() ;

 

 //

 // Critical Section:

//   此時中斷不會產生，所以全域變數A與不會被ISR破壞

 //

 B = 10 ;

 A = B + 5 ;

 

 // 恢復中斷

 //

 enable_interrupt() ;

 ...

}

 

 

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

// Function: 中斷處理函數

//     中斷可能在任何程式執行時產生

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

__interrupt__ void ISR(void)

{

 // 通常ISR執行時中斷都會被禁能

 //

 ...

 

 // 在ISR內操作全域變數

 //

 B = 12 ;

 ...

}

 

菜鳥：『C語言程式會對應到多行的組合語言指令，而這些組合語言指令會廣泛的使用內部暫存器，就像上個例子的R0一樣；既然全域變數要保護，這些暫存器也應該要保護吧，否則ISR執行完畢後，所有暫存器的值都變掉了，返回被中斷的程式後一定會有問題吧？ˊˊ用台　』

 

韌體老鳥：『沒錯，因為暫存器會一直被使用，所以不能用禁止中斷產生的方法。ISR寫作要注意的第二個事項為：ISR的第一個動作一定是將所有的CPU內部暫存器儲存起來，一般都是存在stack memory內，而ISR的最後一個動作就是回覆所有暫存器的值。所以當程式自中斷返回繼續執行時，所有暫存器的值會和發生中斷前完全一樣，不會因為中斷發生而影響程式執行的正確性。通常CPU會有類似“pushall”與“popall”的指令，意指將所有內部暫存器暫存於stack memory中，如果沒有push-all與pop-all的指令，我們就要逐一將所有暫存器push到stack memory中，並在ISR的最後以相反的順序將所有暫存器值pop出來。以下是一個範例：器行時動作就是指令之間都可能產生中斷。我舉個極端的例子好了，ˊˊ用台　』

 

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

// Function: 中斷處理函數

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

__interrupt__ void ISR(void)

{

 // 儲存所有暫存器到stack memory中

 //

 asm("push R0")

 asm("push R1")

 ...

 asm("push R14")

 asm("push R15")

 

 // ISR程式本體

 ...

 

 // 取回所有暫存器的值

 //

 asm("pop R15") ;

 asm("pop R14") ;

 ...

 asm("pop R2") ;

 asm("pop R1") ;

}

 

菜鳥：『所以這樣推演下去，在ISR中呼叫其他函數也要很小心囉，如果在ISR中呼叫的函數也會操作到全域變數，也可能造成全域變數被破壞的狀況。ˊˊ用台　』

 

韌體老鳥：『沒錯，而且這種狀況更難查，因為有的函數是來自其他廠商或單位的函式庫，一來我們可能沒source code，就算有，我們也不熟其運作原理，發生的問題時根本無從查起。所以ISR寫作的第三個注意事項是：呼叫外部函數時要確認其是否“可重進入”(Reentry)[1]，如果一定要在ISR內呼叫非reentry的函數，則一般程式在使用同一個模組的函數時，一定要加以保護。

 

ISR內呼叫外部函數的問題比較複雜，特別是在我們不熟這些函數的設計原理時；前面我們談的只是一兩個全域變數所造成的影響，當我們在ISR內呼叫了一個函數，我們是否真的可以確認這個函數中用了幾個全域變數，以及這些全域變數又被其他哪些函數使用？如果無法確認的話，則一般函式中使用到“可疑”的函數時都要保護。ˊˊ用台　』

 

菜鳥：『其實最保險的方法就是盡量不要在ISR內使用未保證為reentry的函數，但這樣一路討論下來，我們得到盡量不要用全域變數與函數的結論，此來ISR內可以做的事情豈不是很少？ˊˊ用台　』

 

韌體老鳥：『ISR內的程式是越簡單越好，ISR寫作的第四個注意事項是：ISR的程式或執行時間盡量不要太長，一來不易掌握全域變數以及外部函數的使用，二來ISR執行時通常不允許其他中斷產生，如果ISR執行時間太長，亦即上層程式被暫停的時間過長，使用者會覺得系統的效能很不穩定。

 

最後，我會建議ISR與其他程式的系統架構為：ISR盡量不要牽涉到複雜的演算法，它的任務只是負責弄清楚發生了什麼硬體事件(舉例來說，哪個鍵被按下、轉換完的AD值為何、哪個timer已經到時…等)，然後將硬體事件送給上層的程式，交由系統或應用程式決定要如何處理這個硬體事件。具體的作法是ISR會將硬體事件的細節儲存在諸如message-queue的資料結構中，而上層程式會從message-queue中取得硬體事件的資訊並做合適的處理。ˊˊ用台　』

 

菜鳥：『在這種系統架構下，message-queue是ISR與其他程式的唯一橋樑，顯然所有操作message-queue的程式段都是必須保護的critical section，但整個系統也只有message-queue相關的函數需要保護而已；這樣的架構使得ISR的內容只是單純的硬體操作，至於該如何處理硬體事件則是上層系統或應用程式的事情(例如“使用者按了PLAY鍵，系統會開始播放MP3”，驅動程式只要負責告知上層程式PLAY鍵被按下了即可)。ˊˊ用台　』

l  名詞定義 - Reentry：可重入的程式碼指的是一段可以被多個任務(task或ISR)同時呼叫，而不必擔心資料會被破壞的程式碼（比如：一個函數）。亦即可重入型函數在任何時候都可以被中斷執行，過一段時間以後又可以繼續運行，而不會因為在函數中斷的時候被其他的任務重新呼叫，而影響函數中的資料。

 

 

 


[1] 所謂的reentry的函數就是當其執行到一半時，發生中斷，或所在的task被暫停並切換到其他的task，即使此函數在ISR或其他task中又被呼叫，當返回繼續執行時也不會影響結果的正確性。根據本節的討論，基本上盡量不要用到全域變數或呼叫其他會用到全域變數的函數，即可保證該函數是可重進入的。
