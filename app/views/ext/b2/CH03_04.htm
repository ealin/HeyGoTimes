3.4.  標準C Library可以用嗎?

 

菜鳥：『我們的硬體平台有LCD，我想要印一些字在LCD上，如果LCD上有個數字一直在累加，這樣應該也可以證明我的程式正在執行；不知道我們的平台有沒有printf()可以用？道如何這台機器才能更新』

 

韌體老鳥：『我換個方向來回答你這個問題，compiler的廠商為了吸引更多開發單位使用他們的產品，當然希望能提供標準C的函式庫給程式設計師使用；如果你是compiler的廠商，你想要怎麼實現printf這個函數？printf()有不定個數參數的問題，比較複雜，你想想如何幫助你的客戶實現print_character()這個函數？只要印出一個字元即可。』

 

菜鳥：『印一個字元還不簡單，先為每一個字元建立一個bitmap，然後只要根據這個bitmap一點一點畫在螢幕就可以show出一個字元了。道如何這台機器才能更新』

 

韌體老鳥：『你的答案有兩個問題：第一，要在“螢幕”上畫點，需要“螢幕”的驅動程式吧，你怎麼知道客戶用的是哪一種“螢幕”？第二，你有沒有想過，客戶的硬體平台是否可能根本沒有你所謂的“螢幕”？』

 

菜鳥：『這…怎麼會這樣，我完全沒想過這樣的事情…道如何這台機器才能更新』

 

韌體老鳥：『所以你還是沒捉到嵌入式系統多樣性的精隨所在，我們可以使用同樣的CPU，同樣的compiler，來開發完全不同性質的產品，例如：PDA和Router這兩種不同性質的產品，就可能使用同一種CPU，用同樣的開發環境開發出來的。對router這樣的網路設備而言，你認為印字要印到哪裡去？』

 

菜鳥：『印到檔案裡嗎？道如何這台機器才能更新』

 

韌體老鳥：『印到檔案是一個選擇，透過網路傳給某一台裝有特定接收程式的PC，也是一種print out的方式；但問題是，compiler廠商怎麼知道你的硬體平台或應用是什麼！還有一個很容易明白的例子，compiler廠商要如何實作動態記憶體配置函數 – malloc()/free()？他們根本不知道客戶的硬體版子中記憶體的狀況，包含起始位址、大小…等等。

 

所以針對這些和硬體相關的標準C函數，compiler廠商就各出奇招，以malloc/free為例，通常會提供幾個非標準的函數，讓程式設計師設定記憶體區塊的資訊後才可正常使用malloc/free；但每家compiler廠商的作法可能都不一樣。為了避免不確定性，通常我們都會自己實現print out以及記憶體配置等功能；一來容易維護，二來比較容易根據我們產品的需求做調整，特別在做效能調整時，這些底層的函數常常是瓶頸所在。

 

此外標準C中還包含了一系列檔案操作的函數(fopen、fread、fwrite、fclose…)，檔案系統的複雜度比列印或記憶體管理高的多，所以不管compiler廠商支援到什麼程度，我們還傾向用自己的檔案系統比較有把握。』

 

菜鳥：『所以像字串操作(strcpy、strcat…)這些和硬體無關的標準C函數就可以直接用囉。道如何這台機器才能更新』

 

韌體老鳥：『是的，只要你觀念清楚，什麼可以用，什麼不能用應該很容易想得出來；不過你不用擔心，compiler使用手冊裡都會清楚標明使用函式庫時的注意事項。這和PC程式開發有很大的不同，嵌入式系統初學者一定要把這個觀念弄清楚。』

 

菜鳥：『我想我搞清楚狀況了，如果我要實現像printf()這樣的函數，我至少要有我們平台的LCD驅動程式，也許團隊中或CPU廠商已經有人做好了，我可以拿來直接用；如果沒有人做過，那我就只能自己來，compiler提供的函式庫是幫不上忙的。道如何這台機器才能更新』

 

 

=============================================================

 

本節提到了標準C-Library，筆者想花點篇幅繼續深入探討；以下以memset() or memcpy()為例，說明嵌入式系統標準函式庫的效能考量。

 

現代程式開發工具的便利性與上個世紀相比簡直不可同日而語，在嵌入式系統開發領域亦是如此；尤其在compiler與library的領域，因為GNU C/C++即其附屬函式庫品質日趨成熟，加上其開放原始碼與容易移植的特性(移植compiler到新的CPU還是有相當程度的困難與複雜度，但GNU C/C++ compiler已成功移植到許多平台，相關技術文件與可參考的實例相當豐富)，所以現在幾乎所有新的CPU幾乎都會以GNU C/C++套件(tool-chain)當作其開發工具的基礎，輔以GDB當作遠端除錯工具，再以免費或相當便宜的價格提供給開發商。

 

這對工程師而言算是一個好消息，一來使用不同CPU開發產品時對開發工具的學習曲線減緩，而且GNU C/C++ tool-chain附屬的標準函式庫使得程式更容易移植到不同的平台上(其實這點帶來的好處比想像多，試想網路上有多少open source的project是基於這些標準函式庫的？這意味著這些共同技術資產都可能無痛地移植到您的專案上)；再加上這些函式庫都是open source的，開發者可以根據產品需求以及平台的特性自行修改之。

 

但使用諸如ANSI C、POSIX…等函式庫還是得小心，舉例來說，在PC上printf()自然是輸出到螢幕上，但在嵌入式系統上可就不一定了，有的平台連“較明顯”的輸出設備(如LCD)都沒有；再例如malloc()/free()，函式庫的開發者可麼可能會知道每一個產品的記憶體起始位置與大小？也就是說不是所有標準函式庫都可以直接用在每個平台上。

 

另一個應該注意的事項是效能問題；每個平台或CPU都有其特性，在許多應用上，要開發出在所有平台皆具有優良效能的演算法是不切實際的；再加上這些函式庫最開始還是以PC當作平台，正確性最重要，演算法的效能並非最重要考量，PC上的CPU自然可以將其執行得飛快。

 

以下舉一個實務上常常碰到的問題 – memcpy(memory copy) and memset(memory setting)：

 

相信讀者對這兩個函數都不陌生，有的應用需要大量的記憶體搬移，此時memcpy()的效能就很重要；舉例來說，一般要在LCD上顯示字元或圖畫都是採用frame buffer，亦即在記憶體中配置一塊video RAM，LCD上每一點都會對應到特定的記憶體空間，而且連續的點(X軸)在video RAM中的記憶體位置也是連續的；每一個點所佔據的記憶體空間則依目前設定之最大色階而定，如果是黑白系統，則一個bit代表一個點，16色灰階，則4個bit代表一個點，依此類推。

 

所以只要設定video RAM中某一點所代表的記憶體位置，LCD controller在下次refresh時就會將這個改變反應到LCD之上，從而達到在LCD“畫圖”的效果。在這種架構中，要完成一個圖形必須一點一點逐一修改video RAM內容，如果要畫的圖比較複雜，或者必須邊解壓縮邊畫，則會讓user感覺到圖畫好像一點一點或一行一行慢慢show出來，這種感覺相當不好；user寧願等待一小段時間，然後畫面一瞬間全部刷新。

 

所以實務上我們都會採用double-buffer的做法，顧名思義就是兩塊video RAM，一個代表前景(LCD controller在refresh時會將代表前景之video RAM的內容反應到LCD panel上)，另一個則是背景； 繪圖函式庫會將圖畫在背景(改變背景video RAM的內容)，等到所有的繪圖工作告一段落後，再一股腦的將整個背景video RAM(或者只包含此次改變的區域)的內容複製到前景的video RAM中，等下次refresh時，user就會發現LCD panel的內容改變了。(有的LCD controller提供隨時改變video RAM位址的功能，在上述應用中就無須做記憶體搬移)

 

現在問題來了，在搬移整個video RAM的過程中，如果效能不彰，對user而言也會感覺整個螢幕是由上至下、一行一行的改變，此來使用背景video RAM等於沒有效果；我們可以試算一下，一個320x240x16bit、70Hz(每秒刷新70次)的panel，其video RAM的size為：320x240x2 = 153600Byte，約150KB的資料量要在小於1/70秒內複製完畢，這使得開發者必須採用該平台上效能較佳的記憶體搬移演算法。

 

最簡單的記憶體搬移方式是呼叫標準C函數 - memcpy()，但這個函數的效能如何呢？每個平台有其最佳的access記憶體的方式，例如從偶數位址一次搬移兩個byte或四個byte；但當初寫memcpy()的工程師並不知道您的平台特性為何，也不能保證您傳入的一定是偶數位址，所以其實現方式相當簡單明瞭：

 

void memcpy(void * dest, void * src, int n)

{

 // 這個function也有可能是用assembly實現的, 但算法基本上一樣

 //

 int i ;

 char *Dest = (char *)dest ;

 char *Src = (char *)src ;

 

 for(i=0;i<n;i++)

        Dest[i] = Src[i] ;

}

 

可想而知這是效能最差的做法，在大部分16bit or 32bit的CPU上，在奇數位址上access一個byte是最浪費CPU運算資源的事情；以上述應用為例，假設您使用的平台在當位址為四的倍數，同時access四個byte的效能最佳，是否可以把前、後景video RAM的起始位址都設在四的倍數的地方，然後呼叫自己的memory copy function？

 

void my_memcpy(void * dest, void * src, int n)

{

 // for Video-RAM copy only

 //

 int i ;

 long *Dest = (long *)dest ;

 long *Src = (long *)src ;

 

 for(i=0;i<(n>>2);i++)              // 往右移兩個bit è n/4

        Dest[i] = Src[i] ;

}

 

在實務上，使用上面的function速度比原來的memcpy()快上不只四倍，LCD顯示效能不彰的問題也因此解決。

 

如果不能保證傳入的位址都是四的倍數，則可將my_memcpy()作如下改善：

 

void my_memcpy2(void * dest, void * src, int n)

{

 // general purpose memcpy()

 //

 int i ;

 long *Dest = (long *)dest ;

 long *Src = (long *)src ;

 

 if ( ( (unsigned long)Src % 4 == 0) && ((unsigned long)Dest % 4 == 0))

 {

        // 如果Src, Dest都是四的倍數

        //

           for(i=0;i<(n>>2);i++)           

                Dest[i] = Src[i] ;

 }

 else 

      memcpy(dest, src, n);

}

 

實務上一般狀況還是呼叫標準C函數裡的memcpy()，只有在對效能相當敏感的應用時才會去呼叫自己的memory copy函數。讀者可以再對上面的memcpy()函數作進一步的改良，例如這個函數可先作判斷，在起始與結束位址不全為4 byte倍數的狀況下，可先單獨複製前後沒有與4 byte對齊的byte，然後中間大量的資料都是以4 byte為單位作複製；當然這種複製的資料量要夠大，則多作上述判斷對效能提升才是有意義的。

 

這篇牽扯了不少東西，但重點就在使用my_memcpy()的時機與”動機”。Performance調整是每個嵌入式系統工程師的必經歷程，當碰上效能問題時，您可能會後悔當初資料結構、演算法等課沒有認真修；在電子產品開發領域有個很嚴肅的”規範”：不能接受的效能和結果錯誤是等價的，PC上的user可以忍受漏斗轉個半天，但絕對無法忍受電子雞變成慢動作。
