4.6.     Source Tree設計

 

菜鳥：『我負責的模組已經寫完並且經過基本的單元測試，我該如何把它整合到我們的系統去？我們明外域變數統中況的多吧』

 

專案經理：『我們不是有一份source tree設計文件嗎，哪些模組要放在哪一個目錄下，裡面規劃的很清楚；當你加入新的程式後，記得寫個mail通知負責系統整合的同仁，他會更改mekefile；然後你最好再花個時間檢查一下，不要讓系統build不過，也不要因為你的程式讓系統跑不起來，忙碌中的工程師脾氣可是很差的。的型太複雜˙麼不同式業系統之前入是來的範圍　養如何我開發了一個簡單的時候多讀完了經有域變數統中況的多吧』

 

菜鳥：『我的程式怎麼可能會讓系統跑不起來？我都測試過了耶！我們明外域變數統中況的多吧』

 

專案經理：『這種例子常常發生，你以為OK，但只要整合到系統來就會有問題；原因很多，這裡就不贅述了(讀者請參考第10章)，我舉個簡單的例子給你看：的型太複雜˙麼不同式業系統之前入是來的範圍　養如何我開發了一個簡單的時候多讀完了經有域變數統中況的多吧』

 

整合時的重要觀念

 

/**************************************************

    Function: sys_init_all_system_modules

 

 - 所有系統模組初始化的函數統一加在這個函數內

**************************************************/

 

void sys_init_all_system_modules(void)

{

    ...

    sys_init_sys_module1() ;

    sys_init_sys_module2() ;

 

    // 菜鳥: 這是模組“rookie”的初始化函數

    //     它必須在應用程式run起來之前執行

    //

    sys_init_rookie() ;

}

 

 

 

/**************************************************

    Function: sys_init_rookie

 - 模組“rookie”的初始化函數

**************************************************/

void sys_init_rookie(void)

{

 char * buffer ;

 

 #ifdef WIN32

        // Point A : Windows平台執行這裏

        //

       buffer = malloc(20*1024) ;

 #else

        // Point B :我們的平台執行這裏

        //

       buffer = os_malloc(20*1024) ;

 #endif

 

 // 在初始化階段不可能要不到記憶體,

 // 如果真的要不到記憶體，要讓系統停下來，否則繼續往下執行一定會出問題

 //

 if(buffer == NULL)

    system_halt() ;              // 系統停止往下執行

}

程式4-8：新模組整合到系統時可能會出問題

 

專案經理：『如果模組設計的好，它的獨立性應該很強，在其他系統模組還沒穩定或完成之前，通常都可以先在Windows的開發環境(如Visual Studio)上作，除了可以在穩定的環境上開發外，程式除錯也比較方便；相信你也是這麼做的吧。

 

以上面這個例子來說，模組獨立開發時會執行Point A，基本上向Windows要求配置動態記憶體很少會失敗的，所以這個程式在Windows上可以通過測試。但整合到開發中的系統可就不同了，系統會執行Point B向系統要求配置記憶體；可能在sys_init_rookie()執行前，其他的程式就把記憶體要光了、也很可能os_malloc()根本就還有問題、甚至可能記憶體驅動程式還沒完成，所以系統執行到sys_init_rookie()就會停下來。此時你就會聽到辦公室中幹聲四起，因為所有人的系統都執行不了了。的型太複雜˙麼不同式業系統之前入是來的範圍　養如何我開發了一個簡單的時候多讀完了經有域變數統中況的多吧』

 

菜鳥：『可是這是別的模組的問題不是嗎？我們明外域變數統中況的多吧』

 

專案經理：『是別的模組的問題沒錯，但卻是因為你加入你的程式才導致目前系統不能運作的，而這個動作浪費了團隊中大部分人的時間也是不爭的事實。這是觀念問題，團隊中的每一個工程師在整合某個新的程式到系統中之前，都必須先從server[1]上抓下目前最新版本的系統，在自己的電腦上測試後，確認不會影響其他人，才可以將這個程式加入server。的型太複雜˙麼不同式業系統之前入是來的範圍　養如何我開發了一個簡單的時候多讀完了經有域變數統中況的多吧』

 

菜鳥：『如果發生問題時應該通知誰？難道沒問題的程式，因為其他先加入的模組有問題就不能加入嗎？我們明外域變數統中況的多吧』

 

專案經理：『假設原來的系統是可以動的，因為加入你的程式而不能動了，那麼誰最適合來查這個問題？應該就是你吧，即便最後查出不是你程式的問題，那你應該通知部門主管，越早抓到隱藏的問題越容易解決，這對研發團隊而言是大功一件啊。

 

此外，所有的程式都應該接受版本控制；所以我的建議是工程師還是要把程式加入server，並且先自行記錄這個問題，稍微修改一下程式讓系統可以繼續執行，例如先mark掉sys_init_all_system_modules()中的sys_init_rookie()，等系統穩定後再回來做該模組的整合測試。的型太複雜˙麼不同式業系統之前入是來的範圍　養如何我開發了一個簡單的時候多讀完了經有域變數統中況的多吧』

 

 

 

 

Source Tree設計原則

菜鳥：『話說回來，我們系統的目錄結構是誰負責設計？用什麼原則設計的？我們明外域變數統中況的多吧』

 

專案經理：『基本上應該由能夠綜觀全局的人來設計，目前的目錄架構幾乎是沿用以前的專案，最早是本人在當資深工程師時設計的，有什麼意見嗎？的型太複雜˙麼不同式業系統之前入是來的範圍　養如何我開發了一個簡單的時候多讀完了經有域變數統中況的多吧』

 

菜鳥：『意見是不敢有啦，我只是想知道設計目錄架構的原則是什麼？整個專案那麼多的程式檔，要如何井然有序的安排？目錄結構太複雜似乎也不好，一堆檔案擠在同一個目錄下，就像沒整理一樣也不好。我們明外域變數統中況的多吧』

 

專案經理：『你只要想想嵌入式系統的特性，你就知道該怎麼擺了。的型太複雜˙麼不同式業系統之前入是來的範圍　養如何我開發了一個簡單的時候多讀完了經有域變數統中況的多吧』

 

菜鳥：『這個之前你說過，嵌入式系統軟體首重可移植性、硬體相關與硬體無關的code要分開、產品相關與產品無關的code要分開等等。

 

這樣我有點概念了，你的意思是說，好的嵌入式系統程式的目錄架構應該可以很容易在其他的專案中重複使用；假如下個產品的硬體平台不變，只有應用程式改變，則系統與driver的程式目錄就可以直接拿來用。再假如下個產品的硬體被換掉，但系統與應用程式只需小幅調整，則只需把驅動程式的目錄替換掉即可。我們明外域變數統中況的多吧』

 

專案經理：『沒錯，觀念就是這麼簡單，但在實務上往往沒有辦法這麼達到這麼理想，總會有人把程式放到不合適的目錄，不然就是在硬體無關的模組裡寫入和硬體相關的程式碼；研發主管只有不斷用心地code review才能避免這種不受規範的程式產生。偏偏到專案後期每個人都忙得很，而且人都是短視的，總想著無論如何先讓目前的專案可以結案再說。所以在設計階段就該把source tree建構好，相關規定越早發佈越容易施行，因為此時完成的程式還不多，所以工程師也比較願意配合。

 

而且我們source tree的結構並不複雜，應該看了就知道每一個目錄是做什麼的，基本上一個目錄可以對應到系統架構圖裡的一個方塊；以下稍微解釋一下目前專案的目錄架構[2]：的型太複雜˙麼不同式業系統之前入是來的範圍　養如何我開發了一個簡單的時候多讀完了經有域變數統中況的多吧』

 

l   Driver –        驅動程式目錄

n   Boot-Loader – Boot-Loader程式目錄

n   Hardware – 各個裝置的驅動程式

u  Include – 驅動程式的header file

u  API – 將所有驅動程式包裝成API

u  Boot – Boot程式

u  LCD – LCD程式目錄

u  SDRAM – SDRAM的驅動程式

u  …

l   System – 系統程式目錄

n   Include – 系統程式的header file

n   API – 將所有系統程式包裝成API

n   Common – 系統程式中的通用功能

n   RTOS – 嵌入式作業系統目錄

u  Include – RTOS模組的header File

u  API – 將RTOS的功能包裝成API

u  Task – task或thread管理

u  Memory – 計憶體管理

u  Sync – task間同步機制

u  IPC – task間通訊機制

u  …

n   Glib – 圖形函式庫

n   Sub-System: 各個子系統目錄

u  GUI – 圖形使用者介面子系統

u  TCPIP – TCPIP Stack子系統

u  FileSystem – 檔案系統

l   AP – 應用程式目錄

n   Include – 應用程式的header file

n   Source – 應用程式的程式碼

u  Common -應用程式中的通用功能

u  AP1 – 應用程式1的目錄

u  AP2 – 應用程式2的目錄

u  …

n   Resource –統一管理應用程式中的圖形、字串等資源檔

l   Third_Party_Library – 由協力廠商或網路取得的函式庫

n   HandWriting – 手寫辨識函式庫

n   Font – 字形

n   聲音壓縮函式庫

l   Include – 專案的header file，sys_config.h(系統組態)就放在這個目錄

l   Build –所有用以製作系統執行檔的檔案，如makefile以及link script

l   Simulator – 模擬器目錄

n   Build – 模擬器的build目錄，可以產生模擬器執行檔

n   Simulated Hardware – 在PC上模擬硬體設備

u  LCD – 模擬LCD的程式

u  Keyboard – 模擬keyboard的程式

u  …

l   Tools – 開發過程所需要的工具

n   Cross Compiler – CPU廠商建議的編譯器

n   Offline Tools – 開發過程會用到的工具，可能由團隊自行開發

l   Document - 和專案有關的文件

n   Project Management – 和專案管理有關的文件圖表，如schedule等

n   Design Spec – 所有設計文件

n   Hardware – 硬體相關的文件，如線路圖、晶片的data sheet等

n   Others – 其他技術文件或sample code

 

 

============================================================

 

上述source tree的結構只是一個範例而已，讀者應該依專案特性與需求自行訂定之；重點是訂定專案目錄結構時的觀念，盡量能表達系統架構的精神，不要讓不同性質的程式混雜在一起，否則不但難以管理，此後要移植到其他產品專案或平台也會比較困難。。

 

 

 


[1] 軟體開發的版本控制相當重要，所以團隊中一定會架設一個版本管理的server，有權限的團隊成員都可以登入這個server，取得最新版本或某個較舊版本的程式；並且可以將修改過後的某個程式，提交(稱為check in或commit)到server。這是多人協同開發必備的工具，讀者可參考本書第5.6節。

[2] Source Tree的結構沒有一定的標準，甚至可以說是很主觀的事情；本節的例子僅供讀者參考，實務上應該要依照產品特性與系統架構自行訂定之。
