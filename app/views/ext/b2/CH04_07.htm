4.7.     程式風格典範

 

專案經理：『按照本團隊的慣例，我們會對新進軟體工程師所寫的第一個程式作code review，以確認他的程式風格沒有大問題；每個工程師寫程式的習慣和風格都不同，原本也沒有什麼對錯，但我們是研發團隊，絕對不能任由亂七八糟的程式碼充斥在系統中。正確性、可讀性和可維護性一樣重要，否則當某個藝術派軟體工程師離職後，恐怕沒有人具有一樣的慧根可以改得動他的程式。在我的團隊中，所有軟體工程師都必須遵守一份叫做“程式風格典範 (Programming Style Convention)”的文件。所謂程式風格，不僅是命名規則而已，其中還規範了程式碼中至少應該具有哪些資訊，以及程式寫作的注意事項。』

 

菜鳥：『我是有盡量按照程式風格典範的規定還寫作程式，但我有個不懂的地方 – 我們規範的命名規則很單純，為什麼不直接用匈牙利命名法即可？操作間的關係說工作該如何展開，或測試計畫書應該包含什麼資料。』

 

專案經理：『你確定你真的了解什麼叫“匈牙利命名法”嗎？它到底規定了什麼？恐怕知道的人也不多吧；而且我的經驗告訴我，越複雜的要求，工程師越會陽奉陰違。其實一般研發單位用的都是簡化過的匈牙利命名法，重點是它要表達的觀念：讓一個變數名稱除了代表一個變數本身之外，更重要的是讓檢閱程式的人能夠對該變數的意義一目了然。我們當然可以依我們的實際狀況來更改命名規則；例如我認為得在嵌入式系統中，全域變數(global variable)和區域變數(local variable)的重要性不同[1]，所以我就會規定者兩種變數的命名方法應該有所區別。除了命名規則外，工程師寫作程式必須注意的事項為：域變數統中況的多吧』

 

l   該程式檔的用途描述

l   撰寫者與日期

l   修改履歷

l   用特殊顯眼的符號區分程式區段

l   每一個function必須詳述用途、各個參數的意義、傳回值的意義

l   每一個全域變數必須詳述用途

l   定義資料結構時必須詳述每一個欄位的意義

l   盡可能多寫註解。如果合作單位有外國人，則盡量寫英文，但如果英文表達真的有問題，則寧願用中文寫；否則即便寫了一堆註解，如果沒人看得懂也是白寫。

l   程式中的縮排必須整齊，最好用tab鍵，或直接規定縮排應該要空幾個空格。

l   如果某一個迴圈(for and while loop)或條件式判斷(if)的內容太長，則在結尾大括號( })的後面要加註解，說明這個迴圈或條件式判斷的內容。

l   不要吝嗇空白行

l   一個程式檔的行數不應該太多(筆者認為一個程式檔不應該超過一、兩千行)

 

專案經理：『你不是說你負責的模組已經做好了嗎？那麼我們今天就來review一下你的程式吧。域變數統中況的多吧』

程式4-9：程式風格規範

 

=============================================================

 

軟體開發團隊常常發生這樣的悲劇：接手的工程師完全看不懂離職者留下的程式；更甚者，自己寫的程式，過了兩個月後自己也看不懂了。為了修改其中的bug或新增功能，與其花很長時間把它看懂，工程師會傾向自己重寫。對專案的傷害是必須重複的投入人力與時間在同一件事情上，而且“重工”做出來的模組還必須再測試一次以避免影響其他模組。

 

即便市面有許多功能很強的Editor與程式碼管理工具(例如CodeInsight、Ultra-Editor、Ultra-Compare、WinMerge)，可以做Syntax Highlight或把程式之間的架構與交互關係找出來；但為了程式碼的可讀性與可維護性，工程師的操作間的關係說工作該如何展開，或測試計畫書應該包含什麼資料。程式風格絕對必需被限制，這種限制必須清楚地寫在內部品質文件中，而且專案經理必須命令各部門主管嚴格執行code review，以確定所有工程師都有照規定執行。

 

另一方面，這個限制必須合理、清楚且簡單，不能為工程師帶來太多困擾。本節的例子也許不是最好，但卻是筆者實際運行於嵌入式系統開發專案的實例，讀者可參考之，為自己或團隊量身訂定程式風格典範。

 

 

 

 

Header File

 

專案經理：『不錯！年輕人的可塑性還是比較高；很多資深工程師把遵從這些規定視為痛苦不堪的事情，他們在之前的工作崗位上沒能養成良好的程式寫作習慣，有的甚至以寫作別人看不懂的程式為榮。對專案管理者而言，這些同仁都是team work的漏洞，我必須花很多時間要求小主管去疏通他們的情緒，無論如何都要把這個制度run起來。對了，讓我也看看你這個模組的.h檔』

 

菜鳥：『OK，.h檔要遵循的規範如下：域變數統中況的多吧』

 

l   所有程式檔的規定

l   避免重複include造成的重複定義，基本上header file的開頭和結尾應該是：

 

#ifndef XXX_OS_MSG_QUEQU_H

#define XXX_OS_MSG_QUEQU_H

 

// XXX_OS_MSG_QUEQU_H 是一個其他地方絕對不會用到或定義MACRO

 

// header file的實際內容

…

 

#endif

 

l   常數和巨集(MACRO)定義必須清楚描述

l   定義資料結構(struct、union、enum)和資料型態(typedef)時必須詳述其意義

l   不要在.h檔中實作函數或定義變數，亦即在header file中的是函數或變數的宣告(declaration)，而不是定義(definition)

 

菜鳥：『根據本團隊“程式風格典範”中的規定，每一個模組的header file可以視為該模組簡單的技術文件，也就是說要使用這個模組的同仁，只要看過header file，就應該可以在他的程式中使用這各模組。根據這個要求，我從設計規格書中貼了很多說明到os_message_queue.h中當註解；剛開始我也覺得這種作法麻煩又費時，後來我發現有其他的同仁把我的這個.h檔給印出來當作文件參考，心中有一點莫名的成就感，我覺得這樣才是專業的程式撰寫風格。操作間的關係說工作該如何展開，或測試計畫書應該包含什麼資料。』

 

專案經理：『隨著專案的進行，工程師會慢慢發現，這些規定雖然對他們在開始程式寫作時造成了一些短暫的不便；但到後期除錯與維護時，這些寫在程式裡頭的資訊，會為他們節省可觀的時間。域變數統中況的多吧』

 

 

 

 

 


[1] 全域變數又分為有初值和沒有初值的兩種，存放的空間是在指定的記憶體區域，其位址在linking階段即已決定。而區域變數是存放在stack中，所以每次某function被執行時，其區域變數的位址可能會不同，而且如果區域變數佔據的空間太大，很可能造成stack overflow，會壓到其他用途的記憶體區域。在執行時期，這樣的問題通常不會馬上爆發，所以很容易誤判bug真正的發生原因。嵌入式系統中有關記憶體的注意事項，請參考本書第八章。
