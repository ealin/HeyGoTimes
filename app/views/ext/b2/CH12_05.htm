12.5.            專案結案

菜鳥：『如果出貨不算結案，那什麼是後才能結案呢？我們團隊的原班人馬都已經開始另一個專案的開發工作了，前一個專案的客戶好像還沒有打算和我們結案的意思。板子的們題的地方器，他和』

 

專案經理：『是啊，真是麻煩！我們已經“免費贈送”一個running-change版了，現在這個客戶還想要我們幫忙做cost-down版；不要說我們沒有空，硬體工程師也早就分配到另一個專案去了。我現在請我們公司的業務人員和客戶協調，雖然delay了一個多月，但我們已經確實完成合約中專案範圍內的所有事情；就算客戶暫時不想結案，我想業務人員也會想盡快把錢收回來。

 

因為大部分客戶總是想盡量榨乾我們的最後一點價值，就我的經驗，真正結案都要在開發完成以及產品出貨的N個月之後；當然也有很爽快的客戶，但畢竟還算少數。專案經理總不能等專案成員都已經去做其他專案的事情後，才來收集專案中所有的技術文件、程式、會議紀錄與相關資料吧，屆時會有人理你才怪。為了因應這樣的狀況，我會依循專案管理教科書中把結案分為兩個獨立的步驟：

 

l   行政結案：對研發團隊來說，專案的經驗累積相當重要，所以專案經理一定要趁key man都還在的時候進行行政結案；所謂的行政結案最重要的工作就是收集並整理所有的文件、資料與檢討報告，我的作法是當產品出貨後的一個禮拜，所有大小主管什麼事都不用做，就是和我關在會議室中收集資料和檢討此次專案的得失，至於其他基層同仁則“在營休假”，只要配合他的主管收集資料即可。當所有資料收集完成、製作清單並備份之後，接下來才可以分批解散專案成員，歸還專案資源，如電腦、量測儀器…等，所有文件歸檔之後才算完成行政結案作業。

 

以嵌入式系統來說，行政結案時至少收集到以下資料：

 

n   技術文件之外的所有的專案基本文件，包含；合約複本、專案計畫書、預算書、成本分析、schedule、WBS(work breakdown structure)、人員組織圖…等。

n   產品規格書及其修改紀錄

n   硬體相關圖面，如迴路圖、layout圖…等，最好從target board到最終量產的版本都有。

n   硬體設計規格書以及所有零件的data sheet。

n   自零件廠商處取得之所有技術文件與sample code。

n   所有與ID以及機構設計有關的圖面與文件

n   每一個版本的備份光碟

n   量產版本的完整程式碼

n   版本管理系統之資料庫備份

n   bug管理系統之資料庫備份與各時期的bug狀態報表

n   所有與專案有關的mail與會議紀錄

n   專案檢討報告

 

l   合約結案：如果專案發起人，通常是客戶，願意進行合約結案的話，我們會安排正式的close meeting，並把行政結案階段收集的所有資料準備好，然後“恭候”客戶代表前來驗收。在這個會議中，雙方可以討論專案進行期間的缺失，以及客戶對最終交付成果的看法；有些客戶會要求對尚未解決的open issue或bug逐一解釋，再次確認對產品沒有嚴重影響後就會將其一一close掉。依據客戶類型的不同，close meeting有時很快就可結束，有時要費時N天；有時只要專案經理或老闆在就可以，有時則需要工程師列席備詢。此時我們通常會逆來順受，只求客戶趕快在結案同意書上簽名，並確認取得尾款的日期，最後還要請客戶填寫滿意度調查表；合約結案完成後，這個專案到此正式結束。

 

專案管理的教科書告訴我們，應該要先完成合約結案，才可以進行行政結案。但對嵌入式系統開發專案而言，實務上真的很難等到合約結案，才開始做專案資料的收集與人員解散；屆時所有人早被分配到其他的工作上，再把他們集合起來做行政結案也只是虛應故事罷了。

 

當然我也碰過半路腰斬的案子，通常是客戶主動放棄委託我們開發；原因很多，可能是因為開發單位技術能力不行或時程嚴重delay，或客戶發現產品定義錯誤或已過時，甚至是文化不和或合作不愉快…等等都有可能。此時就只剩下合約、開發費、甚至賠償的問題，雙方的法務人員自然會去協調；研發團隊的所有同仁當然會相當沮喪，但這也是沒有辦法的事情，提前結束總比“不會結束”好吧。ˊˊ用台　』
