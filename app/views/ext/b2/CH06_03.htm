載入data段

 

菜鳥：『我認為原理很簡單，就是“有設定初值的全域變數”必須被儲存在執行檔裡、被燒錄到ROM裡，但執行時因為這些全域變數的值可能會被程式改變，當然不能在ROM裡運行，連結時期必須重定址到RAM裡；就是這種“儲存在ROM，在RAM執行”的特性，所以才有搬移data段的需求，而且必須在所有程式使用全域變數之前完成這件事。

 

我覺得比較麻煩的是LMA與VMA的定義[1]，我老是會把它們搞混；此外，在link script中定義可以在程式中使用的變數名稱，對我個人來說，這的確是令人印象深刻的新概念。責人行程式境』

 

韌體老鳥：『慣於寫作PC程式的工程師應該會認為“全域變數可以被設定初值”是天經地義的事情吧，殊不知這是因為Windows或Linux上的編譯工具幫每一個程式加了稱為startup code的程式碼；startup code就像boot-loader一樣，負責為使用者的程式佈置合理的執行環境，其中當然包含配置stack、heap的空間，以及設定全域變數的初值等。PC程式可以這麼做是因為PC的CPU可以實現虛擬記憶體，每一個程式有自己的位址空間，所以每一個程式中各個區段(程式段、資料段、bss段等)的起始位址可以是一樣的，這就是為什麼編譯工具，可以提供所有使用者程式通用的startup code。

 

嵌入式系統的開發者可就沒那麼好命了，每一個產品的記憶體配置可能都大不相同，所以編譯器廠商當然無法提供通用的startup code，開發者只有自己寫一途。煩了軟體、還是工具的問題都不知道，變數一堆要怎麼分析問題何在。官狀況是題何在知道責人行程式境』

 

菜鳥：『這個做法我知道，首先在link script中定義data區段的定址資訊時，並需順便定義幾個變數，以下面的link script片段為例[2]，這些變數的意義為：

 

l   __START_data_LMA：data段在ROM中的位址(可把LMA想為儲存位址)

l   __SIZE_DATA：data段的size，亦即要搬移的byte數

l   __START_data：data段在RAM中的位址(可把VMA想為執行位址)

 

有了這三個變數，我想搬移data段的程式就很容易了。看程式之前，我們還是複習一下執行時期的記憶體使用狀況。責人行程式境』

 

圖6-6：執行時期的記憶體使用狀況

 

 

    /* ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

本例中data段的VMA跟在bss段之後，而LMA則跟在rodata段之後

意義為： data段的內容會被加入執行檔內，位置在rodata之後

                    但執行時期data段會被複製到RAM中，位置則在bss段之後

~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~*/

    .data __END_bss : AT( __END_rodata )

      {

            __START_data = .;

            *(.data) ;

            __END_data = .;

      }

 

/*        定義可在程式中使用的變數 – “__START_data_LMA”，

表示data段的起始位址(LMA)  */

__START_data_LMA = LOADADDR( .data );

 

 

/*   定義可在程式中使用的變數 – “__SIZE_DATA”，

表示data段的size */

__SIZE_DATA = __END_data - __START_data ;

 

程式6-6：link scrip範例 – data段描述

 

 

韌體老鳥：『嗯，觀念和程式都很簡單；對初學者可能會覺得寫link script麻煩了一點，把語法弄懂之後，會發現其實蠻單純的。再來就是LMA和VMA的觀念不要搞混，總之記得前者是儲存位址，後者是執行位址就不會弄錯了。』

 

菜鳥：『之前我看到的範例是一個byte一個byte搬，後來我翻CPU的data sheet發現這樣的效能很差，我們目前使用的是32 bit的CPU，外部的data bus和CPU內部的一般暫存器都是32 bit，所以對CPU而言，一次操作4個byte的效能最佳；這個搬移的程式應該這麼寫：責人行程式境』

 

// 定義在link script中的變數，這裡直接宣告為指向unsigned long的指標

//

extern unsigned long * __START_data ;

extern unsigned long * __START_data_LMA ;

 

extern int __SIZE_DATA ;

 

void copy_data_section(void)

{

 int i ;

 unsigned long * dest = __START_data ;

 unsigned long * src =  __START_data_LMA ;

 

 // dest和src指到的位址，以4個byte為一個單位

 //

 for(i=0;i<(__SIZE_TEXT_SPEED_UP/4);i++)

    dest[i] = src[i] ;

 

}

設定bss段

韌體老鳥：『bss段的設定相對來講比較簡單，因為它的成員是沒有初始值的全域變數，所以根本不需要儲存空間，在執行時期只要把bss段的執行空間(VMA)多都設為0即可。』

 

菜鳥：『bss段的處理方式很簡單；以下是link script中有關bss段的敘述，以及boot-loader中設定bss段初值的函數：責人行程式境』

 

    /* ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~    

            定義section - bss段， 起始位址(VMA)從0開始

            (假設： 位址0是RAM的起始位址)

~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~*/

程式6-8(A)：link scrip範例 –bss段描述

 

 

extern unsigned long * __START_bss ;

extern int __SIZE_BSS ;

 

void clear_bss_section(void)

{

 int i ;

 unsigned long * dest = __START_bss ;

 

 for(i=0;i<(__SIZE_BSS/4);i++)

        dest[i] = 0 ;

 

}

程式6-8(B)：copy_data_section()快速版

 

韌體老鳥：『在boot階段，data段和bss段是一定要設定的，否則執行時期全域變數的值就不會正確；換句話說，在設定完data段與bss段之前，boot-loader程式是不能使用全域變數的，如果一定要用的話，在變數定義中給定初值是沒有用的，一定要在程式內明確的設定初值才行。請看下例：』

 

// 全域變數定義, 有初值設定的屬於data段, 沒有初值設定的則屬於bss段

//

int global_var_1 = 1234 ;

int global_var_2 ;

 

void boot(void)

{

 inr temp ;

 

 // 此時機器剛剛上電,data段所在之記憶體(VMA)的內容可能為任何值

 // 甚至可能此時這塊記憶體還沒初始化完畢, 根本無法使用

 //

 temp = global_var_1 ;          // 無意義

 

 // 明確的設值後, 變數global_var_1的值才可確認為5678

 //

 global_var_1 = 5678 ;

 temp = global_var_1 ;          // temp = 5678

 

 // 同理, 此時bss段所在的記憶體內容可能為任何值

 //

 temp = global_var_2 ;          // 無意義

 

 // 設定data段

 copy_data_section() ;

 

 // 將bss段全部清為0

 clear_bss_section() ;

 

 temp = global_var_1 ;          // temp = 1234

 

 temp = global_var_2 ;          // temp = 0

…

}

載入程式段

菜鳥：『除了data與bss段外，需要加速的程式模組也需要在boot-loader階段搬移嗎？還是讓使用這個程式模組的程式自己做就可以了[3]。』

 

韌體老鳥：『這是一個好問題；你也知道要搬移程式區段時需要用到link script中定義的變數，讓應用程式工程師負責這個事情可能比較不恰當，一來應用程式工程師可能對相關知識比較不熟悉，而且萬一搬到錯誤的位址就麻煩了。

 

通常我們會希望link script由負責架構開發環境的同仁負責統一管理，他應該是團隊中最了解當前系統記憶體配置的人，要把什麼區段搬到哪裡他最清楚，所以一直以來我們團隊的作法都是讓他處理程式段搬移的事情。至於要是否要在boot階段做則視情況而定，如果要搬移的是重要的系統程式模組，則儘早就定位會比較好；如果要搬移的是應用程式模組，則我們會寫一個函數讓應用程式在需要的時候呼叫。有時候實際狀況會比較複雜，例如不同程式模組要共用一塊記憶體區塊，在這種狀況當然是需要時才做載入的動作。』

 

菜鳥：『會有多個模組共用一塊記憶體的狀況！那link script不就更複雜了嗎？』

 

韌體老鳥：『你要知道一個觀念，越快的記憶體肯定越貴，越貴的資源在系統裡自然就越缺乏；我們在學校修計算機組織時都學過一個觀念，每單位記憶體的價格從最貴到最便宜的排行為：CPU暫存器 > CPU Cache > CPU內部記憶體 > 外部RAM > 硬碟，這個觀念在嵌入式系統中依然成立，我們可以把上面的關係再細分為：

 

CPU暫存器 > CPU Cache > CPU內部記憶體(internal) > 外部SRAM > NOR Flash[4] > SDRAM > Mask ROM > NAND Flash[5]

 

你可以想像在系統中，工程師可以使用的ROM一定比速度較快且昂貴的SRAM大的多，這也是為什麼CPU的internal RAM總是小的可憐，像目前我們這顆CPU的internal RAM就只有8KB。所以系統中需要加速模組無法同時載入到internal RAM中執行，只好將各個模組都定址到同一個位址，視需求再載入執行。』

菜鳥：『PC上隨便寫一個Windows的應用程式就要1M byte多，8K byte能做什麼？』

 

韌體老鳥：『那是因為編譯工具幫你連結了許多函式庫，所以PC程式會那麼大。嵌入式系統的空間要錙銖必較，8KB可以做的事情多了了，估計相當於幾百行C語言程式。尤其到專案後期發現空間不夠時，有可能是RAM不夠，也可能是ROM不夠，你要到各程式中擠出空間來，屆時你就知道在嵌入式系統開發時好的程式寫作習慣有多麼重要。』
