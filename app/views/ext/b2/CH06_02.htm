程式6-1：CPU暫存器操作測試

 

確認Function Call可否正常運行

 

菜鳥：『function call不是C語言的基本功能嗎？而且編譯工具會幫忙完成定址的工作，程式設計師只要寫對function 的名稱，填對參數，該函數就應該可以被正確呼叫執行。我還沒聽誰說過要測試這個的。責人行程式境』

 

韌體老鳥：『Stack記憶體是函數呼叫的基礎，在嵌入式系統開發時，系統程式要自行管理stack，如果管理不當，就可能發生一執行函數呼叫或函數呼叫多幾層後就當機的狀況；而且有許多系統不穩定的現象或無厘頭的bug都是因為stack使用不當所導致。

 

首先你要知道，C語言乃利用stack記憶體完成下列事項：

 

l   function call時的返回位址記錄

l   function call時的參數傳遞(也有些CPU的編譯器在參數個數較少且型態單純時，會用CPU暫存器來傳遞參數；但碰到參數較多或型態複雜的狀況時，一定是利用stack來傳遞參數)

l   函數內部的local變數也是使用stack記憶體

l   ISR被執行時的CPU狀態與返回位址

 

Stack point(目前stack堆積的頂端)的設定與stack記憶體的配置是一件很重要的事情，但很多人都疏忽了。Stack會被忽略的主要的原因是：在Windows或Linux上寫程式時我們完全不需要設置stack的大小與位置，也不太可能發生stack overflow去壓(破壞)到其他記憶體區塊的狀況，這是因為在產生Windows或Linux執行檔時，linker會自動幫程式加上一段startup code，其中就包含了stack記憶體的配置，再加上PC上有虛擬記憶體，所以stack的位址可以與其他程式區段相隔很遠。

 

但在嵌入式系統中就沒有那麼幸運了，linker怎麼可能知道你的記憶體配置？怎麼可能知道要將你的stack設在哪裡？更不要說大部份用於嵌入式系統的CPU都不會有虛擬記憶體的功能。你試著思考一下，在呼叫第一個函數之前boot-loader必須完成哪些事情？個址的錄幸運了叫執行煩了軟體、還是工具的問題都不知道，變數一堆要怎麼分析問題何在。官狀況是題何在知道責人行程式境』

 

菜鳥：『我記得boot-loader的第一行程式就是設定CPU的SP(stack pointer)暫存器，但是我有點搞不清楚C語言的stack和CPU的stack pointer有什麼關係？言的有點搞不清楚責人行程式境』

 

韌體老鳥：『一般工程師的確會弄不清楚這兩者之間的關係；我簡單的說一下吧，當你用C語言呼叫了一個函數，並傳入參數，例如fun(a,b)，編譯後的機器碼應該包含以下動作：

 

a.                     執行CPU指令“push”，將參數a、b存入stack(stack是一種先入後出的資料結構，存入的動作我們稱為push)；CPU執行“push”指令是依據CPU SP暫存器的值，把值存入SP指到的位址，並把SP減一(通常stack是從高位址往低位址堆)的中們稱為吧。

 

b.                     執行CPU指令“call”；CPU會把目前PC暫存器(program counter)的值push到stack中，這個動作即意味著儲存返回位址。

 

c.                      接著CPU把PC暫存器的值設為函數“fun”的位址，則下個被執行的指令就是函數fun的第一行命令。

 

d.                     當函數fun在執行時，可以利用目前SP暫存器的值，計算出參數a與b的位址。

 

e.                     如果函數fun有定義區域變數，這些變數的位址會從目前stack的頂端往下擺；這就是為什麼我們在嵌入式系統開發時，老是會要求應用程式工程師不要定義size太大的區域變數，否則很容易發生stack overflow，請見圖6-2。

 

f.                       當函數fun執行完畢後，CPU會執行“ret”指令(就是return的意思)，這個命令會從stack頂層取出(稱為pop)返回位址，然後更改PC暫存器的值為這個返回位址；則下個指令就會返回執行函數fun的下一行指令，從而完成了函數呼叫

 

以上的每一個動作幾乎都與stack有關，如果SP暫存器沒有設定到正確的位址，或者沒有配置足夠大的記憶體當作stack，那麼程式中去呼叫函數時就很可能出問題。言的有點搞不清楚責人行程式境』

菜鳥：『那麼stack記憶體要放在哪裡比較好呢？而且stack的大小要規劃多少才夠呢？言的有點搞不清楚責人行程式境』

 

韌體老鳥：『其實沒有一定的標準，因為一般CPU的stack都是由高位址往低位址堆，所以在boot-loader階段，我會選擇某塊RAM的頂端當作SP暫存器的初始值，之後驅動程式或系統要重新設定SP就不關我的事了；順帶一提，一般的嵌入式作業系統(RTOS)都會提供多工的功能，每一個執行單位(稱為task或thread)都會有自己的stack，此時就會宣告幾個array當作各個task的stack；請參考以下的程式段：

 

// 定義一個array，當作main task的stack area

               (UW)&(os_main_task_stack_area[2048])

               //

               // 大部分的stack是從高位址往低位址堆, 所以在此必須傳入stack area的頂端

       );     

 

       return MAIN_TSK_ID;

}

程式6-2：用array當作task的stack area

 

 

至於stack的size就更難定義了，在記憶體錙銖必較的系統中，stack size定太大或太小都不對；我們剛開始會定大一點，並規範所有工程師盡量不要定義太大的區域變數，使用遞迴(recursive)時也要小心一點，通常2K~4K byte應該就綽綽有餘了，到後期空間不夠的時候再來調整。至於要調整為多少才合理，最好用工具來量測比較好，簡單的說，就是在每一個function結束前讀取SP暫存器的值，接著讓測試人員操作所有的功能，最後會得出一個stack曾被使用的最大值，把這個值再加一些buffer，應該就是最合適的stack size[2]了。言的有點搞不清楚責人行程式境』

菜鳥：『中斷系統不就是告訴CPU中斷向量表在哪裡就好了嗎？怎麼會有什麼問題。責人行程式境』

 

韌體老鳥：『你把這件事情看的太簡單了，有很多因素可能導致中斷系統不work；換個方向講，boot-loader階段要做好以下工作：

 

a.      中斷向量表陣列(請參考程式3-1)，詳細註解每個entry代表的中斷來源

b.     有的平台是CPU外接中斷控制器，那就必須先完成中斷控制器的驅動程式，才可能開始測試中斷系統是否正常。

f.        提供ISR的寫作範例(請見程式6-2)，讓寫ISR的工程師不用知道中斷系統的細節，基本上就是寫一個C的函數即可。

煩了軟體、還是工具的問題都不知道，變數一堆要怎麼分析問題何在。官狀況是題何在知道責人行程式境』

 

 // ISR模板

// (這只是一個範例，讀者僅需了解ISR該做的事情，無需瞭解程式中的細節)

//

void isr_template(void)

{

    // 將所有general purpose register儲存到stack中

      // (pushn %r15的表示將r0~r15都push到stack中)

     //

    asm("pushn      %r15");

 

      // 將ALR與AHR暫存器透過r1儲存到stack中

      // (讀者無需理會ALR與AHR為何，不同的CPU有不同的暫存器需要儲存)

    //

    asm("ld.w  %r1,%alr");

    asm("ld.w  %r0,%ahr");

    asm("pushn      %r1");

   

      // 呼叫C語言函數，也就是說，工程師只要更改這一行

      // 真正ISR要處理的事情，只要寫在your_ISR()這個C語言函數即可

      //

    asm("xcall your_ISR");

   

      // 從stack中取回ISR被呼叫時ALR與AHR暫存器的值

      //

    asm("popn       %r1");

    asm("ld.w  %alr, %r1");

    asm("ld.w  %ahr, %r0");

 

      // 從stack中取回ISR被呼叫時r0~r15暫存器的值

      //

    asm("popn       %r15");

 

      // 執行“interrupt return”指令，返回被中斷的程式

      //

    asm("reti");   

 

}

記憶體測試

 

菜鳥：『接著記憶體測試是要測什麼啊？不是access的到就好了嗎？責人行程式境』

 

韌體老鳥：『記憶體會出問題的機會比你想像的多；從硬體方面來說，如果資料線或位址線連接錯誤，就會導致讀寫總是錯誤。從軟體方面來說，有些記憶體，如SRAM、NOR Flash與ROM，不需要什麼設定，只要位址給對就可以直接使用。但是其他的記憶體種類，以SDRAM為例，則必須透過額外的控制IC(SDRAM controller)才能操作，程式必須先設定好SDRAM controller的組態，如SDRAM 的size、速度等，才能正確的access到SDRAM；如果某些參數設定錯誤，例如size設定的比較小，則SDRAM可能還是可以動作，只是高位址的記憶體會access不到。

韌體老鳥：『算Checksum是一種簡單且實用的小技巧，望文生義，就是把要驗證的檔案或記憶體區域，一個byte一個byte累加起來，最後會得到的總和就叫做checksum。內容不同的檔案或記憶體區域要算出相同checksum值的機率是非常之低的，程式如下，一個是在PC上計算檔案checksum的小程式，另一個是在機器上計算記憶體checksum的函數，非常之簡單：煩了軟體、還是工具的問題都不知道，變數一堆要怎麼分析問題何在。官狀況是題何在知道責人行程式境』

 

 

/************************************************************

            Program: calculate_checkum.exe

************************************************************/

#include <stdio.h>

 

void main(void)

{

 // 一定要用binary mode開啟檔案

 //

 FILE * fp = fopen("image.bin","rb") ;

 unsigned long checksum = 0 ;

 

 int i ;

 int read_count ;     // 自檔案讀出的byte數

 

 while(1)

 {

    // 把buffer內的值都設為0

    //

    memset(buffer,0,256) ;

 

    read_count = fread(fp,1,256,buffer) ;

    for(i=0;i<read_count;i++)

            checksum += buffer[i] ;

 

    if(read_count < 256)

            break ;

 }

 

 fclose(fp) ;

 

 printf("Checksum of image.bin is : %d", checksum) ;

}

程式6-4：PC上的小工具 - 用以計算image file的checksum

 

/************************************************************

            Function: calculate_ROM_checksum

  - 計算起始位址為0x2000000，size為8MB之記憶體的checksum

************************************************************/

unsigned long calculate_ROM_checksum(void)

{

 unsigned long checksum = 0 ;

 

 // 指到記憶體的起始位置

 //

 unsigned char * pointer = 0x2000000 ;

 

 for(i=0;i<(8*1024*1024);i++)

    checksum += pointer[i] ;

 

 return checksum ;

}
