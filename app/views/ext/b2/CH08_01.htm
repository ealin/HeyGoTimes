CPU + Memory的架構是現代計算機系統的理論基礎(還記得圖靈機Turing Machine理論嗎？)，所以記憶體ˊˊ用台　是嵌入式系統中除了CPU以外最重要的裝置，嵌入式系統中不可能沒有記憶體，否則CPU要到哪裡抓取程式指令來執行？程式中的變數要儲存在哪裡？使記憶體可以被其他程式正確存取，是驅動程式的責任，但配置系統中所有記憶體的用途則是系統工程師的工作；本章的重點為：嵌入式系統開發流程中所有和記憶體有關的細節。

 

 

8.1.     記憶體空間配置

 

菜鳥：『報告學長，又是我，今天起我奉命到系統組“服務”。ˊˊ用台　』

 

系統老鳥：『這幾個月看你在各組間遊走，想必學了不少東西吧；你的運氣算不錯了，很多剛畢業的菜鳥，進到大公司內就被分到某個組內，作著千篇一律的工作，好不容易有機會升到管理職，才發現自己雖然已是某個領域的專家，但對專案管理以及產品開發的視野卻是如此狹隘。工程人員可以訓練，基層管理人才的斷層才是現代企業的最大的隱憂，有時候把資深工程師拔擢為專案經理反而是災難的開端，工程人員和專案經理必須具備的人格特質與處事態度完全不同。ˊˊ用台　』

 

菜鳥：『你不是在說我們PM老大吧？我覺得他還好啊，不但具備工程師的專業技能，對整個產業與專案進度控管的sense也很好啊，至少到目前為止我還蠻服他的。ˊˊ用台　』

 

系統老鳥：『我當然不是在說他，台灣和老美的職場結構不同，在我碰到的美國客戶公司中，四五十歲還在當工程師的狀況在所多有，而且其薪資或地位並不亞於管理職。台灣資訊界的職場生態則是所有資深工程師必須設法爭取管理職，所以我是希望你要自我期許有朝一日可以成為稱職的管理者。

 

好啦，言歸正傳，在開始做系統整合之前，我們必須先弄清楚哪些程式要在哪裡執行？程式可以用哪些區域當緩衝區？程式之外的資料要怎麼儲存…等等；簡單的說就是配置我們系統中所有記憶體的用途，稱之為memory layout，我們就以目前專案為例，你能列出我們的系統中的所有記憶體以及它們的用途嗎？ˊˊ用台　』

 

菜鳥：『據我的了解，我們系統包含以下的記憶體區塊：

 

l   CPU Internal SRAM – CPU內部有兩塊RAM，分別是40K Bytes與16K Bytes：當作LCD控制器的video RAM，或用來加速程式模組的執行效能。

 

l   外部NOR Flash – 1M Bytes：用來放置程式與data，且程式可以在NOR Flash中直接執行；這塊記憶體也可以用Mask ROM取代。

 

l   外部SRAM – 512K Bytes：執行時期放置程式的所有變數(data段與bss段)、程式執行時所需的stack memory、以及動態配置記憶體(dynamic location memory)或緩衝區(buffer)。

 

l   SD Card：我們的系統可支援SD Card，額外的大筆資料可以儲存在SD Card中。

 

這樣的架構還蠻單純的，我想所有的工程師都知道什麼東西該放在什麼位置吧；例如，程式應該要放在ROM或Flash，全域變數以及buffer應該放在RAM，這些想當然爾的事情怎麼可能會有人搞混？ˊˊ用台　』

 

系統老鳥：『什麼東西要放在哪一個記憶體是一回事，“要怎麼放進去”又是另外一回事；例如NOR的內容要透過燒錄器寫入資料或者update程式更新，SRAM則只需要知道位址就可以把資料寫入，至於SD Card則必須透過card file system來存取其中的檔案，不同記憶體操作的方式可能都不同，一般工程師可能會搞不清楚。而且話說回來，就算是最簡單的SRAM好了，你怎麼知道要用什麼位址去存取它？ˊˊ用台　』

 

菜鳥：『這個我在寫驅動程式時曾經學過，先從線路圖或硬體設計文件得知CPU是用哪根Chip Select腳與SRAM連接，然後再查CPU的data sheet就可以知道這根Chip Select腳連接之記憶體的起始位址。 ˊˊ用台　』

 

系統老鳥：『是啊，你知道這些細節，你覺得應用程式工程師會知道嗎？或者說，他們需要知道嗎？應用程式可以直接定義全域變數或陣列，至於這些變數會被定址到哪兒去則會由系統工程師負責的link script決定；但有時系統或應用程式可能暫時需要一塊size比較大的buffer，工程師必須要知道系統中哪個位址空間的記憶體是可以使用的，此時他們也只需要知道位址範圍即可；所以系統人員必須提供一張簡單易懂的圖或表格，可以讓所有參與專案的工程人員都知道系統記憶體配置的狀況。下圖是一個例子[1]：ˊˊ用台　』

系統老鳥：『有了這張圖，所有的工程師就不至於寫出太離譜的程式；例如我們的外部SRAM雖然有512K Bytes，但規劃為可以當作buffer的空間只有128K Bytes，所以應用程式工程師就知道不能用到150K Bytes的buffer，此時要不改變演算法，不然就是和系統工程師討論重新配置記憶體用途的可能性。再舉另一個例子，如果用以加速程式的Internal SRAM在規劃中只剩3K Bytes，應用程式工程師就該把要加速的模組控制在這個size內。

 

基於上述的memory layout，我們來討論一個很常見的狀況，如果你有一個約100K Bytes的資料要加入程式中，你的程式要怎麼寫？ˊˊ用台　』

 

菜鳥：『我們的專案最後會build出一個binary file，燒錄器可將這個binary file燒入NOR Flash(或EEPROM)；所以我們只要把這個100K Bytes的資料加入binary file就OK了，程式執行時它就會在NOR Flash的某個位址上(上圖的Data File區)，利用這個位址我們的程式就可以使用這個100K的資料。ˊˊ用台　』

 

系統老鳥：『基本上大筆資料放在你說的“Data File”區中是比較合理，如果Data File區已經滿了，我們還是可以參酌其他記憶體的使用狀況，把這個大傢伙擺到其他的記憶體區域去：

 

a.                     放置在rodata區(read-only data section)，程式如下；程式中只要用陣列的名稱即可操作這筆資料。

 

const unsigned char my_data[] =

{

        0x2C, 0x02, 0x00, 0x00, 0x0F, 0xB7, 0x43, 0x08, 0x89, 0x44, 0x24, 0x14,

        0xDB, 0x7C, 0x24, 0x08, 0x9B, 0xE9, 0x16, 0x02, 0x00, 0x00, 0x8B,

        0x18, 0x33, 0xC0, 0x89, 0x44, 0x24, 0x1C, 0xDF, 0x6C, 0x24, 0x18,

        0xE9, 0xFB, 0x01, 0x00, 0x00, 0xDF, 0x6B, 0x08, 0xDB, 0x7C, 0x24,

        0x00, 0x00, 0x8B, 0x43, 0x08, 0xE8, 0x8E, 0xFE, 0xFF, 0xFF, 0xDD,

       …

       // 共有100*1024個元素的陣列

       …

        0x02, 0x00, 0x00, 0x8B, 0xC3, 0xE8, 0x86, 0xFD, 0xFF, 0xFF, 0xDD,

        0x01, 0x00, 0x00, 0x8B, 0xD0, 0x66, 0x81, 0xEA, 0x00, 0x01, 0x74,

} ;

 

b.                     如果上述陣列不宣告為const，則這筆資料會被連結到data區(有初值的全域變數)；data區的資料也會佔據執行檔的空間，在執行時期會被複製到RAM中，所以程式可以改變這個陣列的值。這個方法會加快操作這筆資料的效能(RAM速度一定比Flash快)，缺點是執行時期時同一筆資料分別在RAM與Flash佔據一份空間，如果沒有改變內容與加速的需求，這麼做實在是浪費空間。

 

c.                      當NOR Flash或Mask ROM空間所剩不多，但RAM的空間還充裕的時候，有時候我們會考慮將這筆資料壓縮

 

也許你還可以想出不同的狀況、不同的方法，總之我要表達的就是嵌入式系統開發往往要到硬體與成本都已定案後才發現記憶體不足，只是有時缺RAM，有時缺ROM，此時系統工程師就要重新審視目前系統的狀況，設法將程式或資料全部“塞”到我們平台的記憶體去。你要說這是挖東牆補西牆也可以，我們的任務就是在可接受的成本條件下，盡力完成指定的功能。ˊˊ用台　』

 

菜鳥：『我問個外行的問題，為什麼不在硬體設計初期就把系統的記憶體用量估計準確？ˊˊ用台　』

 

系統老鳥：『你都知道這個問題外行了，如果真的可以估計準確，我們為什麼要把自己弄到這個困境中？這個問題可能有三個答案：

 

a.                     真的估不準 – 嵌入式系統開發專案管理有許多事情在專案前期就要進行估計，包含時程、風險、可達成的效能、需要的資源…等等，但這些估計只能憑經驗，很難有更科學的方法。如果專案複雜度低，或者團隊之前做過相似度很高的專案，那麼估計準確的可能性會高一點；否則碰到複雜度高的案子，在專案初期只有初略的產品規格時，要準確估計記憶體用量簡直是不可能的任務。

 

b.                     硬體設計跑在前面，軟體規格卻持續變動 – 硬體規格直接牽涉產品造價，所以通常會盡快定下來；但老闆或客戶總是有錯誤的觀念，認為軟體修該沒有成本，專案期間就不斷恣意變更軟體規格；所以即便專案初期把記憶體用量等估計準確了，到後期已經不是同一回事。

 

c.                      因為成本考量，客戶或老闆幫你估好了記憶體用量 – 這是最無奈的狀況，老闆或客戶當然希望用最少的成本做到最多、最高品質的功能，所以有時候客戶會事先訂下一個離譜的成本底線、或者極為陽春的硬體規格；另外就是天才的老闆或業務人員未與技術人員討論，就對客戶包山包海，答應了一個根本做不到的規格。這樣的案子我碰過不知道多少次了，打從專案一開始就知道記憶體的用量不夠；計算機系統的記憶體容量和效能往往有著密切的關係，當記憶體不夠時，從系統程式到應用程式都要為此付出代價，要不接受較差的效能，要不然就要去研究效能更好的演算法。

 

我想這就是嵌入式系統具有挑戰性的地方；在PC上寫程式，CPU速度那麼快，記憶體那麼大，程式設計師不需要對演算法有什麼研究就可以寫出效能還可以接受的程式，這樣的程式拿到嵌入式系統上執行恐怕會慢的像龜在爬吧。我記得以前學校的老師說過，演算法是計算機科學的精髓之一，而高效率的解題技巧才是一個優秀的軟體工程師該追求的。我想這是嵌入式系統工程師必須經過的磨練，難怪PM老大總是說：我始終相信完整經歷過一個電子產品開發週期的工程師，之後再去參與其他的軟體開發案絕對是駕輕就熟；就像在彈盡援絕的狀況下打過仗的士兵，對不同的環境會有較好的適應能力，此後什麼戰場也難不了他。ˊˊ用台　』

 

菜鳥：『了解，這話我也聽過PM老大講過幾次，現在慢慢可以了解箇中深意了。ˊˊ用台　』
