5.3.  Link Script

 

菜鳥：『製作可執行檔的流程中，要先把所有程式檔都編譯成目的檔，接下來就是連結(link)的動作，透過linker可以將所有的目的檔與函式庫連結為執行檔。無論是否開發嵌入式系統的程式，只要是C或C++寫的程式，執行檔的製作都必須經過這些步驟。我不了解的是為什麼在我們的開發環境中，執行linker的時候要傳入一個副檔名為.ld文字檔[1]，這個檔案比makefile更像天書，內容包含了程式的名稱與一些數字；不就是把所有會用到的目的檔連結在一起就可以了嗎？為什麼還要寫這麼複雜的描述檔？』

 

系統老鳥：『這個文字檔叫做link script，裡面的內容是用來描述專案中的各個程式檔，分別要被定址到哪個位址；link script不像makefile，不同的編譯工具會有不同的語法，想看懂link script恐怕只能詳讀編譯工具的使用者手冊了(雖然不同cross-tools間link script的格式差異極大，但原理大同小異)。實務上我們不會花太多時間研究每套工具的link script語法，通常都只是先拿sample code來修改，碰到特殊需求時再去翻書就好了。

 

現在先不要管link script的語法，你可否先說說看Windows上的應用程式(XXX.EXE)是怎麼執行起來的？和址哪　式』

 

菜鳥：『細節我是不清楚啦，不過在學校修組合語言時老師好像有講過：X86系列的CPU有虛擬記憶體的功能，所以每一個Windows的應用程式有自己的位址空間[2]，程式在作linking時應該都會被定址到某個預設的位址，我們稱這個位址為程式的entry point(進入點)，一般程式設計師根本不用管這個細節。這些細節都被諸如Visual Studio的整合開發環境包裝起來，程式設計師只要告訴IDE專案中有哪些程式檔，IDE就可以把執行檔做出來，至於linking時下了什麼參數知道的人恐怕不多吧。

 

當使用者要執行某個.EXE檔時，系統(此指Windows)會去檢查該執行檔的檔頭，以判斷此檔案是否為合法的執行檔，並取得其相關資訊；然後系統會建立一個新的位址空間，從硬碟中將該執行檔的程式部份載到記憶體中，然後把CPU的執行權跳到該應用程式的entry point，這個程式就可以執行起來了。』

 

系統老鳥：『這個年頭能把這個流程說的頭頭是道的年輕人已經不多了，連結(linking)與執行的觀念就是如此；我把嵌入式系統和Windows應用程式的不同處列舉如下，你再思考一下我們該如何來處理嵌入式系統linking時的工作：

 

a.      系統和所有應用程式通常在同一個位址空間。

b.     嵌入式系統通常沒有硬碟裝置。

c.      程式通常直接在ROM或Flash中執行，而ROM是唯讀的。

d.     程式中的資料段必須儲存在ROM中，因為ROM是唯讀的，所以執行時期必須被載入RAM中。

e.      有時為了執行效率，某些程式段必須在效能比較好的記憶體中執行，亦即不是將所有程式都連結在一起就好了。

f.        最重要的一點，PC上有Windows幫程式設計師處理載入的動作，但在嵌入式系統中，從Reset起的每一行程式都必須自己寫，當然還包含讓應用程式可以順利執行的所有動作。

和址哪　式』

 

菜鳥：『我大概有點概念了；最大的不同點是PC程式完全都在RAM裡面執行，再加上每一個應用程式都有自己的位址空間，而且沒有唯讀的問題，所以所有的程式只要從同一個起始位址連結在一起即可。而嵌入式系統就不同了，程式段可以在唯讀記憶體中執行，但資料段則只能被定址在RAM所在的位址，所以在連結時勢必要告訴linker：程式段要被定址到哪裡(或說ROM的起始位址在哪裡)，資料段要被定址到哪裡(RAM的起始位址在哪裡)。

 

但老實說，即便我已經知道系統中ROM與RAM的位址，我還是完全不知道該怎麼描述“程式該如何連結”這件事情。』

 

系統老鳥：『你已經抓到一個重點，就是在執行時期，程式中的某些區段會被從ROM中搬到RAM，而這些動作不能奢望有個Windows幫你做，只能由我們的系統自行負責。要弄清楚這些概念對初學者的確不容易，因為你必須先知道所謂“程式”的架構。』

 

程式段與資料段

系統老鳥：『細節我就不多說了，資訊本科應該要修“系統程式”吧；一般人只知道程式會分為程式段和資料段，實際上應該再細分為不多說了易執行好了、：：

 

l   text段：就是所謂的程式部分，基本上執行時期時text段的內容不會改變，除非有特別的需求(例如為了加速)，否則text段可以直接在ROM裡執行，無須載入到RAM。

 

l   Read-only-data(rodata)段：程式中定義為“const”的變數，以及諸如“Hello World”這樣的字串，都會被歸類到rodata段(請見以下例子)；程式段還有載到RAM裡執行的可能，但rodata段沒有加速或改變內容的需求，所以應該“待”在ROM裡即可。

 

//

//以下三個變數屬於rodata段

//

const int A = 5 ;

const char B[] = "This is a book" ;

const int A[10] = {11,22,33,44,55,66,77,88,99,0} ;

 

void fun(void)

{

 // “rodata String”會被存在rodata段

 //

 char string[] = "rodata String" ;

 

 // “rodata also!”也會被存在rodata段中

 //

 printf("rodata also!") ;

 

}

 

l   Data段：有初值的global變數都屬於這個區段，請見以下的程式。在連結時期，這些初值必須加入執行檔中；在執行時期，因為這些變數的值是可以被改變的，所以它們不能被定址在ROM的位址，亦即系統在開始使用這些變數之前，必須將其初值載入RAM。所以在連結時，資料段的處理是比較特殊的，一方面它必須被加入執行檔，這樣才可以與執行檔一起被燒入ROM中，但這些變數不能被定址到ROM，它們真正被操作的地方是RAM；總的來說，必須告知linker。data段會被加入ROM，但卻要定址到RAM的位址。

 

 

// 有初值的global變數屬於data段

// 在連結時期, 這些變數的初值會被加入執行檔, 但定址到RAM的位址

// 在執行時期, 這些變數被儲存在ROM中,

// 但必須被載入RAM才可以使用

//

int a = 10 ;

int b[10] = {11,22,33,44,55,66,77,88,99,0} ;

 

 

l   bss(Block Started by Symbol)段：沒有初值的全域變數，請見以下程式。因為沒有初值，所以無需加入程式之中，只要在連結時將其定址到RAM即可；執行時期也沒有載入的問題，但機器RESET後，不能保證RAM的內容都會是0，所以通常要由系統主動將整個bss段清為0。有些工程師寫程式時會自己預設bss變數的初始值應該是0，在未設值的狀況下就直接拿來用，所以每次開機後該變數的初始值可能都不一樣；這個問題呈現出來的狀態就是某功能有時正確，有時錯誤，而錯誤的情形也都不相同，雖然這只是工程師的小小疏忽，但會讓其他人覺得系統很不穩定。

 

 

// 沒有初值的global變數屬於bss段

//

int a ;

char b[10] ;

int buffer[100*1024] ;

 

在嵌入式系統中，上述之程式四種區段的處理方式都不相同，連結時期要藉由link script告知linker如何處理這些區段；在執行時期的系統初始化階段，就要針對各個區段作不同的處置[3]。根據以上的解釋，我們接下來探討link script到底包含了什麼。在此之前，我先考你一個問題：“執行檔”內包含了哪些段的內容？』

 

菜鳥：『這個考不倒我，根據你剛剛的定義，因為bss段是沒有初值的全域變數，所以只要知道它的位址與大小即可，放一堆0在執行檔裡是沒有意義的；而其他的text、data、rodata段都對應到工程師寫下的程式碼，當然必須加到執行檔中。』

 

 

Link Script的內容

 

系統老鳥：『在寫links script之前，我們要先設計執行時期的記憶體使用狀況，下圖是嵌入式系統最基本的記憶體配置：

 

圖5-7(a)：執行時期的記憶體使用狀況

 

首先我們來看一個簡單的例子，你先不要管它的語法，先看註解裡的說明，只要知道這個link script內包含什麼即可：』

 

/************************************************************

                Link Script sample

記憶體位址配置:

ROM的起始位址 - 0xC00000

RAM的起始位址 - 0

 

基本名詞定義：

LMA(Load memory address)：某程式區段被“放置”的位址

VMA(Virtual memory address)：程式區段在執行時期的位址

************************************************************/

 

        /* 定義輸出的格式，本例為：輸出ARM9的機器碼，執行檔格式為ELF32

       */

OUTPUT_FORMAT("elf32-arm9")

OUTPUT_ARCH(arm9)

SEARCH_DIR(.);

 

SECTIONS

{

        /* ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

定義section - 程式段,  起始位址(VMA)從0xC0000開始，

沒有指定LMA，表示LMA的起始位址同VMA，也是0xC00000

~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~*/

        .text 0x00c00000 :

          {

                /*    定義可在程式中使用的變數 – “__START_text” ，

句號 “.”表示目前的VMA，也就是0xC00000

                */

                __START_text = . ;

               

                /*  *(.text)表示將目前目錄中所有檔案的.text段加入這個section*/

                *(.text) ;

 

/* 定義可在程式中使用的變數 – “__END_text” ，目前VMA應該是：0xC00000加上所有程式.text段的size總和

                */

                __END_text = . ;

          }

       

        /* ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~    

                定義section - rodata段

起始位址(VMA)從“__END_text”開始，亦即緊跟在text段之後，

沒有指定LMA，表示LMA的起始位址同VMA

~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~*/

        .rodata __END_text :

          {

                __START_ rodata = . ;

                /*  *(.rodata)表示將目前目錄中所有檔案的.rodata段加入這個section*/

                *(. rodata) ;

           __END_rodata = . ;

          }

 

        /* ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~    

                定義section - bss段， 起始位址(VMA)從0開始

沒有指定LMA，表示LMA的起始位址同VMA

~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~*/

        .bss 0x0000000 :

          {

                __START_bss = .;

                *(.bss) ;

                __END_bss = .;

          }

       

/*定義可在程式中使用的變數 – “__SIZE_BSS”，表示bss段的size

*/

    __SIZE_BSS = __END_bss - __START_bss ;

   

        /* ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~    

                定義section - data段，

                Data段的VMA應該在RAM，而LMA則在ROM

                所以VMA跟在bss段之後，而LMA則跟在rodata段之後

               

意義為： data段的內容會被加入執行檔內，位置在rodata之後

                但執行時期data段會被複製到RAM中，位置則在bss段之後

~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~*/

        .data __END_bss : AT( __END_rodata )

          {

                __START_data = .;

                *(.data) ;

                __END_data = .;

          }

 

/*    定義可在程式中使用的變數 – “__START_data_LMA”，

表示data段的起始位址(LMA)

*/

__START_data_LMA = LOADADDR( .data );

 

 

/*   定義可在程式中使用的變數 – “__SIZE_DATA”，

表示data段的size

*/

__SIZE_DATA = __END_data - __START_data ;

 

}

程式5-8：Link-Script實例

 

系統老鳥：『在解釋這個例子之前，你必須先了解LMA與VMA的意義：

 

l   LMA(Load memory address)：某程式區段被“放置”的位址

 

l   VMA(Virtual memory address)：程式區段在執行時期的位址

 

例如資料段會被放置在ROM，但執行時必需載入到RAM，則在ROM中的位址就稱為LMA，而在RAM中的位址就是VMA。

 

一個程式區段要指定兩種位址(LMA與VMA)，確實是不太容易懂；但是我們之前已經講解過嵌入式系統連結時期的概念，你就試著描述一下上面那段link scrip到底想表達些什麼。』

 

菜鳥：『我再仔細看了一下這個檔案，配合剛剛說的觀念，似乎也沒有那麼複雜。這個link script檔要傳達的概念為：

 

a.                     我們的系統中有一塊ROM，它的起始位址是0xC00000，另外有一塊RAM，起始位址為0

 

b.                     執行檔中必需包含text、rodata與data段，其中text段和rodata當然在ROM裡執行即可，所以被定址到0xC0000，而rodata段跟在text段後面。

 

c.                      bss段因為沒有初值，所以不會佔據執行檔或ROM的空間；它會被定址到RAM的起始位址，在本例中RAM的起始位址為0。式的系統中麼複雜fu6fu6fu6

 

d.                     data段比較複雜，它的內容也必須包含在執行檔內，但執行時期它必須被載入到RAM裡。所以data段的VMA在RAM中，跟在bss段之後；而LMA則跟在rodata段之後，當然是在ROM裡面。』

 

系統老鳥：『我補充一下有關data段的細節， data段會和程式段一起被燒到ROM裡，它的位址(LMA)應該是0xC0000多；但程式中操作這些有初值的全域變數時卻是用另一個位址，也就是所謂的VMA，應該是位址0之上。初學者很容易忽略這個事情，如果在啟動階段沒有將data段載入到VMA的位址，執行時期就會發現這些全域變數的初值都不對，當然會影響執行結果。你再看看下面這張圖應該就更清楚了：』

圖5-7(b)：執行時期的memory mapping

 

系統老鳥：『我再考你一個問題：假設我們有某一個模組(姑且稱之speed_up模組)的效能需要改善，一個可行的方法是讓其在速度較快的RAM中執行，問題來了，請問link script要如何來描述這件事情？』

 

菜鳥：『基本的作法應該和data段一樣，這個模組裡的程式LMA在ROM裡，VMA則定址到RAM中，執行時期在使用這個模組之前，必須先將其從ROM載入到RAM裡；但我不知道詳細的語法怎麼寫。』

 

系統老鳥：『寫法如下(下面的程式段接在程式5-8的最後面)，你要知道其實語法不重要，屆時翻使用手冊即可，重要的是觀念；在嵌入式系統中這樣的應用很常見，通常都是為了改善效能，程式在不同的記憶體中執行有不同的速度，有興趣你可以請教開發韌體的同仁。』

 

 

        /*  speed_up模組的VMA和LMA都是跟在data段之後

speed_up模組會被加到執行檔中，但執行時期要載到RAM才能執行

*/

   .text_speed_up __END_data : AT(__START_data_LMA + SIZEOF( .data ) )

      {

       __START_text_speed_up = . ;

       Speed_up_main.o(.text) ;

       Speed_up_main.o(.rodata) ;

       Speed_up_main.o(.data) ;

       __END_text_speed_up = . ;

 

/* 為容易說明起見，假設這個模組沒有bss段*/

      }

    __START_text_speed_up_lma = LOADADDR( . text_speed_up );

__SIZE_TEXT_SPEED_UP = __END_text_speed_up - __START_text_speed_up ;

 

 

定義在link script中的常數

 

菜鳥：『我還有一個實作上的問題，剛剛我們說要把data段從ROM搬到RAM，用說的很簡單，但是我們程式中怎麼知道data段在ROM的位址和size？怎麼知道要搬到哪個位址？難道是要用寫死的嗎？而且在上面的那個例子中，data段在RAM中的位址跟在bss段後，所以每次bss段大小更動，data段的起始位址(VMA)也會變動，而且data段的大小也可能變化，那豈不是每次都要去改動程式？會不會太麻煩了？』

 

系統老鳥：『以前的編譯工具功能比較陽春，確實有可能要如你所說把位址寫死在程式中，是很麻煩沒錯。但現代linker的功能比較強，除了提供某些預設的變數名稱 (如“__start_text”表示程式段的起始位址(VMA)等)，工程師可以在程式中直接使用外；一般linker都可以讓工程師在link script中定義代表各區段起始位址(LMA or VMA)以及size的變數，所以我們可以在程式中利用這些變數，達到搬移區段的功能。』

 

菜鳥：『你說得很清楚，我聽得很模糊，可否說得具體點？』

 

系統老鳥：『在剛剛的例子中[4]，我們可以在程式中使用的變數有：

 

l   __START_text_speed_up：speed_up模組程式段VMA的起始位址

l   __END_text_speed_up：speed_up模組程式段VMA的結束位址

l   __START_text_speed_up_lma： speed_up模組程式段LMA的起始位址

l   __SIZE_TEXT_SPEED_UP：speed_up模組程式段的size

 

所以你在程式中該做的事情就是：從位址“__START_text_speed_up_lma”開始(ROM)，複製size為“__SIZE_TEXT_SPEED_UP”bytes的記憶體內容，到起始位址為“__START_text_speed_up”的記憶體(RAM)中。至於程式的寫法為：』

 

//

// 定義在link script中的變數

// (說是常數也並無不可,因為改變這些變數的值是沒有意義的)

//

extern unsigned char * __START_text_speed_up ;

extern unsigned char * __START_text_speed_up_lma ;

extern int __SIZE_TEXT_SPEED_UP ;

 

void copy_data_section(void)

{

 // 一個byte一個byte搬: 這是最簡單, 但效能最差的作法

 //

 int i ;

 unsigned char * dest = __START_text_speed_up ;

 unsigned char * src = __START_text_speed_up_lma ;

 

 for(i=0;i<__SIZE_TEXT_SPEED_UP;i++)

    dest[i] = src[i] ;

 

}

程式5-9：將某個程式模組搬移到速度較快的記憶體上執行

 

菜鳥：『真神奇，想都沒想過居然可以在程式之外定義變數。』

 

系統老鳥：『給你一個作業，想想看如何在程式中把整個bss段的內容設為0。』

 

=============================================================

 

清除bss段比搬移程式區段簡單的多，請參考以下程式：

 

// 請參考本節中完整的link script

//

extern unsigned char * __START_bss ;

extern int __SIZE_BSS ;

 

void clear_bss_section(void)

{

 int i ;

 unsigned char * dest = __START_bss ;

 

 for(i=0;i<__SIZE_BSS;i++)

    dest[i] = 0 ;

 

}

 

Map File或Symbol Table

 

系統老鳥：『連結後除了產生執行檔外，通常還可要求linker產生map檔(例如GNU linker “ld”的-m參數)；雖然每套編譯工具產生的map檔格式可能都不一樣，但它的基本用途就是紀錄專案中每一個symbol[5]之LMA與VMA的對應關係。這個map檔很有用，我們可以從map檔中得知下列訊息：

 

l   程式各區段的定址是否正確

l   程式各區段的size，亦即目前記憶體(ROM與RAM)的使用量

l   程式中各個symbol的位址

l   各個symbol在記憶體中的順序關係(這在除錯時很有用)

l   各個程式檔的記憶體用量

 

節錄一段map檔的內容來看看你就會明白了：』

 

=============================================================

因為map檔不是程式，所以實際上不會有註解在裡面；以下範例中用/**/包起來的文字，是筆者自行加入的解釋。

=============================================================

 

/*

text段資訊： VMA = 0xC00000、在執行檔中的起始位址=0、LMA = 0xC00000

                        程式段的VMA和LMA是一樣的

*/

.text                0xC00000              0x0          load address 0xC00000  

 

/* “__START_text”是在link script中定義的變數 */

0xC00000                              __START_text=.

 

/*

程式Module.o的text段資訊： 起始位址為0xC00000，text段占0x300個bytes

*/

Module1.o(.text)

 .text                      0x0C00000      0x300             Module1.o

 

/* Module.o中的函數有三個，分別列出其起始位址 */

0x0C00000                                      fun1

                0x0C00200                                      fun2

                0x0C00280                                      fun3

Module2.o(.text)

 .text                      0x0C00300            0x100             Module2.o

                0x0C00300                                    fun3

                0x0C00380                                    fun4

…

…

/*

若有使用到函式庫內的函數，則該函數所在的程式檔也會被連結到執行檔中

*/

.text                        0x0C01000           0x100             ..\..\lib\os.a(mem.o)

                0x0C01000                                   localMalloc

                0x0C01080                                    localFree

…

                0x0C02000                                   __END_text=.

 

…

/* RAM起始位址為0，假設bss段size = 0x100, rodata size = 0x1000 */

 

/*

資料段的資訊：VMA = 0x100、在執行檔中的位置 = 0x3000、 LMA = 0xC03000

 

由此可以看出VMA和LMA的位址不一樣，VMA指到的位址是RAM，data段跟在bss段之後；LMA指到的位址是RAM，data段跟在rodata段之後

*/

.data                       0x000100    0x3000         load address 0x0C03000

0x000100                          __START_data=.

.data                       0x000100    0x2               Module1.o

                0x000100                          global_short

 

/*

Midule2.o中data段的資訊，可以看出data段的起始位址(VMA)為0x102；其中有兩個有初值的全域變數，共佔8個byte。

*/

.data                       0x000102      0x8                  Midule2.o

                0x000102                             global_int_1

                0x000106                             global_int_2

…

               0x000200                __END_data=.

 

 

菜鳥：『這個檔案裡面全是數字，看得人眼睛都花了；我怎麼有辦法從裡面找到我要的資訊？』

 

系統老鳥：『作出執行檔之後的下個步驟就是設法將其下載到機器上，作出執行檔是一回事，你怎麼知道你的程式段或資料段會不會太大，會不會超過ROM或RAM的size？你怎麼知道你的link script有沒有寫錯，每一個程式區段都確實定址到符合機器的記憶體設定？當然你可以說在機器執行一下不就知道了嗎，可是我認為負責整合的工程師一定要檢查一下map檔，有些問題只會造成系統的不穩定，而不會馬上當機，這種問題最麻煩了。

 

例如資料段超過RAM的size，在測試時可能不會馬上出問題，只有在用到高位址的資料時才可能會壓到其他變數；即便破壞了其他變數，只要系統不會馬上用到這個被破壞變數，那當下也不會感覺程式有問題；當發現系統出問題時，極有可能已經在不相關的功能上了，這對除錯會造成很大的困擾，一定要盡量預防。

 

至於檢查的方法很簡單，只要搜尋各個區段的起始和結束位址，以及你自己在link script定義的變數，確認其LMA、VMA和size都沒有問題就可以了。』

 

 


[1] 不同cross-tools裡的linker可接受之link-script的格式可能不同，本書是以GNU的linker – ld為例，其他linker的link script格式基本上與此大同小異。

[2] 位址空間(Address Space) – 較高檔的CPU都會提供虛擬記憶體的功能，有了這個功能，系統可以實現讓各個應用程式宛如有自己的記憶體空間(稱為位址空間)，就算在自己的記憶體空間中亂搞，最多就是自己當掉，不會影響到系統以及其他正在運行中的程式。虛擬記憶體的功能可用來實現多工、多使用者系統，但複雜度相對來說比較高。而一般電子產品並未有此需求，所以大部分用於功能較單純之嵌入式系統開發的CPU都不會有此功能，亦即系統與所有應用程式會共用一個位址空間。

[3] 本節旨在說明連結時期的概念，有關執行時期對各個區段的特別處理，請參考第6.3節。

[4] 每個linker的語法都不一樣，讀者只需要在此學到觀念即可；實務應用時還是要以實際使用的linker使用手冊為準。

[5] Symbol表示程式中所有出現過的函數、函式庫、global變數名稱，以及linker自動產生之代表各個區段(text、rodata、data、bss)起始結束位址的變數。
