7.2.     準備工作

 

菜鳥：『我迫不及待要開始寫驅動程式了，我該從哪裡開始？我們系統中有LCD、有keyboard、有touch panel、還有一堆不知道幹什麼的IC，我想都應該要寫程式去驅動它們吧，你就分一個簡單一點的讓我試試看好了。ˊˊ用台　』

 

韌體老鳥：『你什麼準備工作都沒做，你想要怎麼開始？驅動程式開發的工作模式和天馬行空想一個演算法來解決某個問題完全不同，即便沒有上述作業系統的限制，硬體設計和周邊裝置都已經先定了下來，你總需要對你要寫驅動程式的硬體設計有所了解吧，否則你怎麼會知道CPU和週邊設備是怎麼連接的，如果不知道怎麼連接的，你驅動程式要怎麼寫？你總要知道我們選用的週邊設備或IC要如何驅動吧，如果你不知道要下什麼命令才能讓某一顆IC初始化，你驅動程式要怎麼寫？

 

所以驅動程式在整個系統開發中是比較沒機會發揮個人創意的工作，若CPU或周邊IC規定就是要照某些步驟執行它才能正常運作，你就不可能用其他方法讓它動；這是很容易理解的道理，你要用它的功能，就得按照它的規定來走。ˊˊ用台　』

 

菜鳥：『這我理解，那麼在開始驅動程式開發之前要先準備些什麼？ˊˊ用台　』

 

檢視可用的資源

韌體老鳥：『首先你必須思考寫驅動程式需要什麼人或物的支援，在開始之前你必須先弄清楚，在開發期間你是否能夠取得這些資源與協助。以下是嵌入式系統驅動程式開發應該設法取得並妥善應用的資源：ˊˊ用台　』

 

l   IC原廠的技術人員或代理商的FAE：晶片的代理商會為其代理的每個產品提供一個技術資源的窗口，我們稱為FAE；因為這個技術支援的人力是由代理商指派，所以應該都是本國人，不會有語言上的障礙。照理說，FAE應該是最熟悉其負責之晶片的技術人員(實際上鬼混的FAE也不在少數)，所以一旦我們選擇了某顆晶片用於我們的產品，無論硬體或韌體工程師，一定要設法熟識原廠或代理商的技術人員，特別是CPU。在開發期間，無論是設計、實現或除錯階段，我們都可以從廠商的技術人員身上得到適時的協助。

 

l   評估板：不只CPU有評估板，週邊IC的廠商也會提供評估板；這些可以運作的板子，對硬體工程師而言是設計階段最好的參考，對韌體工程師來說則是自己的硬體平台尚未完成前的實驗平台。此外評估板也可以在硬體除錯階段當作對照組之用。

 

l   Sample Code：韌體工程師應該善用廠商提供的sample code，如果可以的話，直接用在系統中也沒有關係；因為這些sample code應該是由最懂這些CPU或IC的技術人員寫的，我們自己慢慢研究也不會比他們更清楚這些IC，寫出來的驅動程式也不見得會比廠商提供的更完整。

 

l   Data Sheet：data sheet是硬體人員設計線路與韌體工程師撰寫驅動程式最主要的依據，有些CPU的data sheet厚達幾百頁，產品開發的時程是不允許工程師慢慢的、仔細的研究其來龍去脈；基本上data sheet的讀法應該是先快速的全部瀏覽過一次，然後挑重點詳讀(有些部份是寫給硬體工程師看的，韌體工程師就可以先略過)，最好能搭配sample code一起看更好。

 

 

l   硬體板子設計規格書：前面說過，最好韌體工程師可以全面參與硬體的設計；如果做不到的話，至少要參加重要的硬體設計審查會議，除了線路圖、layout圖之外，硬體設計單位應該要提供設計規格書、CPU各個PIN腳的使用規劃表格等書面資料；韌體工程師必須依據這些資料規劃驅動程式設計工作，如果有不可行的硬體設計，一定要儘早與硬體工程師溝通。

 

l   線路圖與layout圖：韌體工程師至少要能看懂線路圖中CPU是用哪些PIN腳與各個外部IC連接的，這直接牽涉到驅動程式該如何控制這些IC。至於layout圖，韌體工程師則通常只要能找到線路圖的零件在板子的哪個位置即可，在量訊號時才不會量錯位置。

 

菜鳥：『我們公司也不是什麼大公司，那些IC廠商的技術人員會鳥我們嗎？ˊˊ用台　』

 

韌體老鳥：『這就要靠我們的口才了，這些賣晶片的廠商的確是“唯量是問”，如果是你也會選擇優先服務“量”比較大的客戶吧；所以身為小廠商的我們，有時候是得吹牛一下，誰知道你最後的產品可以賣幾台？在不離譜的前提下把這次產品的量吹大一點；不然就是狐假虎威，把我們的大客戶抬出來，暗示IC廠商若不好好服務，我們的客戶以後可能就會把該廠商列為觀察名單或拒絕往來戶。經過如此威迫利誘，IC廠商的技術人員通常都會乖乖來我們實驗室報到。如果這幾招都行不通，乾脆就我們一群人，帶著notebook和板子直接殺去他們公司找他…或他的主管。ˊˊ用台　』

 

菜鳥：『想不到資訊界是如此黑暗…ˊˊ用台　』

 

韌體老鳥：『IC的代理商設置FAE這個職務就是為了服務使用他們零件的客戶們，通常FAE一個人要負責許多公司代理的零件，他要服務的公司與單位很多，你總得給他一個理由讓他優先服務你吧。ˊˊ用台　』

 

用C語言寫驅動程式

 

韌體老鳥：『另外一個對驅動程式的誤解就是：驅動程式通常都要用組合語言寫，這也是讓一般工程師對驅動程式開發懷有恐懼感的原因之一有恐懼感的原因之ㄧ。但實際狀況是無論Windows、Linux或嵌入式系統的驅動程式開發，幾乎都是用C語言為主，除非必要我們盡量不會去寫組合語言程式。而且現在的C編譯器通常都支援inline assembly的語法，所以原則上我們不會去寫一個全部都是組合語言的程式檔(.s或.asm)，需要組合語言的時候在.C檔中插入inline assembly即可。ˊˊ用台　』

 

菜鳥：『是啊，一開始我也是對用C語言寫驅動程式完全沒概念，上次在評估板上寫作LED閃爍的程式後，我才知道在C語言中可以使用以下的技巧來開發驅動程式：ˊˊ用台　』

 

 

l   Volatile變數：操作CPU內memory mapping register的方法就是使用C語言的指標，為避免相關程式被最佳化(例如連續三行程式 “*a = 1 ;”，就會被C compiler的最佳化功能變成一行)，必須將指標形態設為“volatile”，請見下例：

 

volatile unsigned int * xxx_register = 0x300024 ;

 

// 0x300021、0x300020、0x200024是CPU的memory mapping Register

// 以下的運算是將bit 4設為1

//

        *(volatile unsigned char *)0x300021 |= 0x10;     

        *(volatile unsigned char *)0x300020 |= 0x10;     

 

        // 對暫存器0x300024連續設值，因為變數形態設定為volatile

        // 所以對xxx_register的操作不會被最佳化

        //

        *xxx_register = 0x00ABCDEF ;

        *xxx_register = 0x12345678 ; ;

 

 

l   Inline Assembly：在現代的嵌入式系統開發，為了可移植性、可維護性等原因，除了要操作CPU內部暫存器以及對某個模組做效能調整外，用組合語言寫程式的機會不多；通常我們也會用inline assembly，免去組合語言函式與C語言函式間互相呼叫時的麻煩。請見以下實例：

 

// 在C語言函式中寫組合語言程式

//

void boot(void)

{

    //  %sp與%psr都是CPU的內部暫存器，

// 用C語言的標準語法無法對其設定

    //

    asm("xld.w  %r4,0x3fff"); 

    asm("ld.w   %sp,%r4");

 

    asm("ld.w   %r8, 0b10000");    

    asm("ld.w   %psr, %r8");  

    …

}

 

l   中斷處理程式(ISR)的寫法：雖說ISR其實就是一個C的函數，但ISR的起始和結尾與一般的函數還是有點不同，簡單的說，在ISR開始時要將所有暫存器都儲存在stack memory中，結束前要恢復這些暫存器的值，而且一般函數是用“return”的指令返回呼叫的函式，而ISR必須用“interrupt return”的指令返回被中斷的程式[1]。原本這些細節都要寫ISR的工程師自己寫，有的編譯器會提供額外的語法，例如在函式宣告時多加一個描述字串“__interrupt__”，則編譯器碰到這樣的函數時，就會自動加上儲存/回覆暫存器的指令，並且用“interrupt return”來返回中斷發生點。請見下例：

 

/*********************************************************

    ISR_01 - 原始的寫法, 必須自行儲存/回存暫存器

                    以及使用不同的返回指令 “reti”

**********************************************************/

void ISR_01(void)

{

    // 將R0~R15儲存到stack

    //

    asm("pushn      %r15");                  

   

    // 透過暫存器R0與R1, 將暫存器ALR與AHR儲存到stack中

    //                   

    asm("ld.w  %r1,%alr");           

    asm("ld.w  %r0,%ahr");

    asm("pushn      %r1");

 

    // 呼叫真正處理中斷的函數

    //

    INT_01_handler() ;

   

    // 回存CPU暫存器ALR與AHR

    //

    asm("popn       %r1");

    asm("ld.w  %alr, %r1");

    asm("ld.w  %ahr, %r0");

   

    // 回存CPU暫存器R0~R15

    //

    asm("popn       %r15");

   

    // 使用與一般函數不同的返回指令 – “return interrupt”

    //

    asm("reti");   

 

}

 

 

/************************************************************

    ISR_02 - 使用了“__interrupt__”描述的函數

            經C編譯器處理後, 產生的組合語言指令與上例相同

            (會自動產生儲存/回存暫存器的指令, 並且用“reti”返回)

*************************************************************/

__interrupt__ void ISR_02(void)

{

    // 呼叫真正處理中斷的函數

    //

    INT_02_handler() ;

}

 

 

l   編譯器定義的暫存器變數：有的編譯器會提供非標準C語言的語法，讓程式設計師可以操作CPU的內部暫存器，並不是所有C編譯器都會提供這種額外的語法，而且我們也不知道使用這種語法時，C編譯器到底會產生什麼樣的code，所以建議還是用inline assembly來操作CPU內部暫存器比較好。

 

 

韌體老鳥：『既然以前已經教過你了，現在我就不花時間贅述了；用C語言開發驅動程式除了程式好寫易讀、容易除錯與維護之外，工程師不用每換一顆CPU就要學習另一種組合語言指令與語法，例如同樣是對暫存器設值的組合語言指令，X86家族CPU[2]組合語言的寫法就與大部分其他CPU的寫法不同。此外驅動程式需要做很嚴謹的單元測試與壓力測試，用C語言寫還是比組合語言方便些。ˊˊ用台　』

 

; X86家族CPU - 對暫存器設值的組合語言語法

; 下例的意義為: 設定暫存器AX的值為 9 (第二個參數搬到第一個參數)

;

mov AX,9

 

 

 

; 其他CPU - 對暫存器設值的組合語言語法

; 下例的意義為: 設定暫存器r1的值為 9 (第一個參數搬到第二個參數)

;

ldw 9,%r1

程式7-2：不同CPU可能會有完全不同的組合語言語法

 

新世紀組合語言實務

 

菜鳥：『如果真的碰到非寫組合語言不可的狀況，例如發現某個模組是系統效能不佳的瓶頸，屆時才去學習這顆CPU組合語言的寫法，會不會來不及啊？ˊˊ用台　』

 

韌體老鳥：『針對你這個問題，我要糾正你兩個觀念，第一，組合語言並不是改善效能的萬靈丹，根據研究，大部分程式效率不佳的原因來自演算法本身的問題，其次是程式寫作習慣，例如用了過多不必要的迴圈或判斷式；和CPU特性有關的主要是資料存取的效能問題，我們前面有提到過，假設CPU操作暫存器或外部記憶體是以32bit為單位，那麼在程式中每次操作1個 byte(char)的效能當然比一次讀出4 bytes(long)到暫存器中的效能差。後斷式迴圈是糾正你兩個觀念言方便些透過改善程式的演算法和多加注意資料存取的效率才是解決效能問題的主要方法，然後調整編譯器最佳化等級，或將該程式模組搬移到效能比較好的RAM裡執行，用組合語言改寫絕對是最後的辦法。以上步驟千萬不能本末倒置，當用組合語言改寫後，再要去找出邏輯與演算法的問題時就沒不像改C語言程式那麼容易了。

 

第二，非得用到組合語言去改寫某程式模組時，通常是對效能要求相當嚴格的應用，例如播放壓縮過後的聲音檔，解壓縮的效能不佳會造成播放時斷斷續續，這樣的效能問題不只是單純的結果比較慢出來而已，而是代表這個應用根本不堪用；這種狀況就必須用上任何的最佳化手段，包含用組合語言改寫。在我的經驗中，我真的不曾從頭到尾寫一個組合語言程式檔，我都是用修改的；有時候要弄清楚某CPU的組合語言函數和C語言函數要怎麼溝通就很令人傷腦筋了，就算弄懂了第一個參數是用第幾號暫存器存放，超過幾個參數就要用stack傳遞…等等細節，等換個編譯器或CPU又有完全不同的規定，工程師的腦力和時間實在不應該浪費在這種事情上。程式的過ˊˊ用台　』

 

菜鳥：『不懂，我們要改寫的是我們自己的程式啊，誰會先幫你寫好再讓你改？ˊˊ用台　』

 

韌體老鳥：『講個真實案例給你聽：在某個案子中，我請硬體廠商的FAE幫我檢查某段組合語言程式的硬體設定部份是否有問題，他後來問我：“你常寫組合語言嗎？這顆CPU我玩了這麼久，也用組合語言寫了不少sample code給其他客戶，但我實在寫不出來像你用了那麼多高級技巧的組合語言程式。”剛剛不是說過嗎，我根本從來不曾從頭至尾寫過一個組合語言程式，那上述的高級技巧到底從何而來？

 

答案很簡單：我不自己寫組合語言程式，但我“請”編譯器幫我作，然後我設法看懂它，並根據我的需求去改動它。ˊˊ用台　』

 

菜鳥：『說的那麼神，編譯器會幫你寫程式？那它以前怎麼不幫我寫論文？ˊˊ用台　』

 

韌體老鳥：『呵呵，別著急嘛，你之前不是研究過開發環境嗎，大部分的編譯器的選項中都有產生組合語言碼的選項(e.g. GNU C Compiler相應的選項是-S)，我都是把要用組合語言寫的程式先用C寫，讓編譯器產生對應的組合語言碼，然後我再拿這個一定正確的組合語言碼來改就好了，以下就是一個C語言函數與自動產生組合語言的對照。ˊˊ用台　』

 

// 原始C語言程式

//

int main( int argc, char *argv[] )

{

        int i ;

       

        for(i=0;i<100;i++)

                printf( "Hello World!\n" );

       

        return 0;

}

;       .loc  3

        pushn     %r0

        ld.w %r0,0x0

__L5:

        .endfile

程式7-3：編譯器自動產生的組合語言碼

 

韌體老鳥：『對一個新的CPU來說，要從頭到尾寫一個組合語言程式出來，大部分的時間都花在遵循許多編譯器或組譯器的規定；還記得我們用組合語言改寫程式的目的為何？不就是為了改善某個模組的效率嗎？如果我們把時間花在學習某CPU組合語言的寫作，豈非本末倒置而且浪費時間嗎？ˊˊ用台　』

程式7-5：驅動程式API設計範例 – Music driver

 

韌體老鳥：『用於播放音樂的硬體設計則有更多的變化，就以播放MP3為例，即便是用軟體解[3]，CPU輸出的訊號還是要經過DAC IC(Digital to Analog Converter)、OP(放大器)以及靜音迴路等等；如果用硬體解的話，則CPU需要透過某種介面與MP3解碼IC溝通，目前業界常用的介面是I2C或I2S [4]，此時驅動程式中就要去實現這些protocol…你在我們設計的Music API中看得到這些細節嗎？

 

假設我們用的MP3 IC的型號是KME-3333，將Music API中播放音樂的函數定義為KME3333_play()是錯誤的，一來這樣的介面勢必會引起應用程式工程師的恐慌，二來此後我們如果換了其他IC，甚至可能採用MP3以外的格式，硬體迴路設計可能完全不一樣，此來KME3333_play()這個介面顯然就不合適了，那就意味著應用程式必須跟著修改。所以我們把播放音樂的API定義為drvMusicPlay()，對應用程式工程師而言，其代表意義非常清楚，基本上他們只需知道call了這個function後，硬體就會開始播放音樂，至於硬體真正的運作方式為何則是驅動程式工程師該關心的事情。跟迴路是哪一科前好ˊˊ用台　』

 

菜鳥：『所以當產品規格定義完成，以及硬體架構大致底定後，Driver API就可以開始進行設計了。我想在設計這套API時，應該要將重心放在系統式與應用程式的需求上；對於硬體設計，我們只需確認其確實能夠達成這些API的功能，至於硬體迴路的細節應該不會影響Driver API的設計。跟迴路是哪一科前好ˊˊ用台　』

 

韌體老鳥：『沒錯，實務上我們會先把.h檔和包含空函式的.c檔先寫好，這樣的好處是：在驅動程式還沒穩定之前不會影響系統的編譯，其他的程式都可以同步開發；此來我們可以會接收到其他同仁對Driver API的意見，有必要的話也可以儘早修改設計。

 

此外我們一般稱Driver API為硬體抽象層(HAL、Hardware Abstract Level)，它對嵌入式系統開發還有一個重要的意義：因為Driver API上層的程式都是和硬體無關的，所以負責模擬器的同仁只要在PC上模擬出這些API的行為，其他上層的程式應該完全不需要任何更動，就可以在機器與模擬器上運作，這樣的架構自然會讓系統有較佳的可移植性。跟迴路是哪一科前好ˊˊ用台　』

 

軟硬之爭-這板子真的可以動嗎?

菜鳥：『記得你說過在我們自己的板子完成之前，驅動程式工程師必須先在CPU廠商提供的評估板上搭配sample code進行開發；我的問題是：評估板除了CPU外，搭配的零件和我們的硬體設計可能會不一樣，在這種情況下，驅動程式工程師在板子完成之前要怎麼做準備工作？跟迴路是哪一科前好ˊˊ用台　』

 

韌體老鳥：『我們先把狀況簡化一點，假設CPU廠商的評估板是採用零件A，而我們的硬體設計將採用零件B。我們可以試著從硬體工程師的角度來看這件事情，他當然可以依據零件B的特性，參考評估板上CPU與零件A的線路設計，再依循專業知識與經驗來規劃我們板子的迴路設計；因為PCB板子製作是一個蠻花費時間與金錢的步驟，如果在請PCB廠商開始製作板子之前都沒做過實際的實驗，這似乎是太冒險了一點。這好像軟體工程師對已經穩定的系統做了一些修改，即便這個修改很簡單，在未經測試的情況下就發行正式版本是一個風險很大且不負責任的作法。

 

所以硬體工程師在重新製作板子之前，還是會設法評估或驗證設計的正確性，通常都是手工在空的電路板或麵包板上把零件依迴路設計連接，再將這塊子板(dauther board)與CPU評估板連接，必要時要把評估板上的某些零件拆掉；有些IC的廠商就會提供這樣的評估板，硬體工程師可以設法修改，再把它和CPU的評估板連接。此一手工打造的平台就可用以驗證硬體設計有沒有大問題，但硬體工程師也僅只能驗證其電氣特性，還是需要韌體工程師幫忙寫程式驗證。跟迴路是哪一科前好ˊˊ用台　』

a.      LED電源供給有問題(硬體問題)

b.     連接LED子板後，導致評估板無法運作(比較像是硬體問題，通常是電源問題、接錯PIN或焊接技術不佳…)

[1] 一般函數呼叫時，CPU只會將返回位址存在stack中，所以指令“return”只會從stack中取得返回位址；但中斷發生時，CPU除了會將返回位址儲存在stack中，還會將PSR暫存器也儲存到stack裡，所以當ISR結束時就必須用另外一個指令(interrupt return)返回中斷發生點，不能用“return”指令返回，否則stack中會殘留一個值，此後運作都會出問題。

[2] 所謂X86系列CPU泛指用於PC上的80386、80486、Pentium、P2、P3、P4…的CPU，其實X86 CPU家族還有一個80186，指令集與基本的X86 CPU相容，廣泛使用於較低階的嵌入式系統。

[3] 讀者可以想像PC上的WinAmp或foobar這類的播放器，它們是用純軟體去解MP3，這需要極大量的數學運算，在PC上當然不成問題，但在嵌入式系統的CPU上恐怕就心有餘而力不足了。所以市場上有許多解MP3的IC，CPU只要將MP3檔案的內容設法傳給這顆IC，它會負責解碼的動作；這種架構我們稱為硬體解。

[4]讀者可先不理會這裏提到的技術名詞，我們的重點是driver API必須將這些硬體特性都包裝起來。

[5] 這是一個很有警惕性的案例，我們在本節的後面還會再詳細說明這個案例。
