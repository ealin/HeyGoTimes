執行檔要下載到ROM(通常我們說“燒”到ROM執行)並直接在ROM中執行，這應該沒有問題吧；但其他的檔案呢？因為嵌入式系統通常沒有硬碟這樣的裝置[1]，所以必須把專案中所有會用到的檔案(包含可執行檔的binary file)，結合成一個下載工具可接受的單一二元檔，我們通常稱這個動作為make ROM，這樣的檔案我們稱為image file，研發團隊在發佈版本時也應該是image file，而不只是單純的執行檔或其二元檔。』

以下是一個簡單的PC程式，可以將PC上的某個檔案，轉成C array：

 

/*******************************************************************

    Tool – Bin2C.exe

            將xxx.jpg轉為C array，並輸出到xxx.c中

*******************************************************************/

#include <STDIO.H>

 

main()

{

 FILE * binfp;

 FILE * cfp ;

 unsigned char buffer[1024] ;

 int byte_read = 0 ;

 int i ;

 

 // 讀者可自行更改輸入，輸出檔名

 //

 binfp = fopen("xxx.jpg","rb") ;

 cfp = fopen("xxx.c","wt") ;

               

 fprintf(cfp,"//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~\n") ;

 fprintf(cfp,"//\t\tBMP Image as C Array\n") ;

 fprintf(cfp,"//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~\n") ;

 fprintf(cfp,"//This File is created automaticly \nunsigned char NAME[] = \n{\n") ;

 

 do

 {

            // 讀出xxx.jpg的內容，並利用fprintf()的功能，將每個byte轉為文字

            //

        byte_read = fread(buffer,1,1024,binfp) ;

        for(i=0;i<byte_read;i++)

        {

                if(buffer[i] <= 0xf)

                       fprintf(cfp,"0x%x ,",buffer[i] );

                else    fprintf(cfp,"0x%x,",buffer[i] );

 

                if(i%16 == 15)

                        fprintf(cfp,"\n\t") ;

        }

 

 } while(byte_read == 1024) ;

 

 fprintf(cfp,"\n} ;\n//end of file") ;

 

 fclose(binfp) ;

 fclose(cfp) ;

 

}

File System Binary

 

系統老鳥：『首先我們來看一下維基百科中對檔案系統的定義：』

 

電腦的檔案系統是一種存儲和組織電腦檔案和數據的方法，它使得對檔案的訪問和查找變得容易。檔案系統通常使用硬碟和光碟這樣的存儲設備，並維護檔案在設備中的物理位置。但是，實際上檔案系統也可能僅僅是一種訪問數據的界面而已，實際的數據可以是通過網路協議提供的或者在記憶體上，甚至可能根本不存在對應的檔案（如 proc檔案系統）。

 

嚴格地說， 檔案系統是一套實現了數據的存儲、分級組織、訪問和獲取等操作的抽象數據類型。

 

 

菜鳥：『簡單的說檔案系統就是將檔案儲存、操作等細節抽象化的程式，它會提供一套簡單的介面(API)讓其他程式使用。至於檔案儲存在什麼地方、用什麼格式儲存則都有可能，重點是使用檔案系統的工程師根本不需要知道這些細節。所以即便嵌入式系統沒有硬碟或光碟，還是有辦法實現檔案系統；但是我還是想不出來怎麼在ROM上面實現檔案系統，ROM不是唯讀的嗎？』

 

系統老鳥：『你還是執著於硬碟的概念，CD與DVD不也是為唯讀的嗎，Windows是怎麼實現檔案系統的？一般嵌入式系統需要的檔案系統其實很簡單，因為執行時期通常沒有寫入檔案的需求，所以只要在其他程式要求讀取某個檔案時能夠知道該檔案所在的位置即可。不像用於硬碟上的檔案系統(諸如FAT32、NTFS，或Linux上的ext2/ext3)，一個檔案可能會被切成多個部分來儲存，我們的檔案系統中的每一個檔案都是連續的，所以嵌入式系統的檔案系統表格相當單純，只要記錄每一個檔案的檔名、起始位址和size即可。下圖是一個用於嵌入式系統檔案系統的概念，非常簡單：』

菜鳥：『所以所謂的檔案系統充其量只是一個索引表格罷了；設計系統時要把先把檔案系統表格的格式定義好，然後負責開發環境的人要設計一個工具，在製作image file時要去掃描所有要加入系統的資料檔，這些檔案要如上圖般一一跟在執行檔的binary file之後，所以可以算出各個檔案在image file的位置(offset)，根據這些資訊可產生file system table；然後把可執行檔的binary file、所有資料檔以及file system table逐一加到image file中。這個過程應該就是之前說過的making ROM吧。』

=============================================================

 

因為電子產品的功能越來越複雜，所以越來越多的嵌入式系統必須使用檔案系統。實務上可以採用本節描述的方法，此外幾乎所有目前業界在使用的嵌入式作業系統，都會額外提供file system的功能(稱為middleware)，例如μCOSII就可搭配μCFS；開發團隊可以直接使用，無須自己重新開發。

 

但這些file system middleware支援的檔案系統格式通常都是業界標準的格式，例如FAT16；在這種情況下，研發團隊必須評估是否真的需要用到FAT，以及對系統效能的衝擊等議題。筆者的經驗是：除非應用程式有頻繁寫入檔案的需求、產品必須可以接受使用者插入自己的記憶卡(使用者自己的記憶卡通常都是標準FAT的格式)、或者產品可以用USB與PC連接變成U-Disk…否則使用FAT的意義不大，而且對系統的overhead很高。

 

 


[1] 隨著NAND Flash的普及與降價，現在很多嵌入式系統都會把NAND Flash當作類似硬碟的儲存裝置；即便是使用NAND Flash，在大量生產時，RD也要提供binary file(或說image file)供燒錄器大量燒錄。

[2] 實務上會自行開發一個工具，把某個目錄下的所有檔案一一轉為C array的程式檔，並產生一個.h檔，裡面包含所有代表資料檔的C array宣告與每一個array的size，工程師只要include這個.h檔就可以使用這些array。
