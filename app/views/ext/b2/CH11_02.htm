非RD人員使用bug管理系統的注意事項

 

菜鳥：『當測試人員把bug單填上Bugzilla後，接下來還需要做哪些追蹤？板子的們題的地方器，他和』

 

測試老鳥：『你已經知道工程師接到新bug時該做的事情了吧，並不是所有的bug都要解，工程師對該bug有任何問題都可以在透過Bugzilla提出，相關人員最好即刻予以回應[3]，可能發生討論的事項為：

 

n   規格的爭議：測試人員與研發工程師對同一功能之規格認知不同，訂定規格的同仁或客戶應立即介入仲裁。

 

n   無須修改的bug：有時測試人員會找出幾近吹毛求疵的bug，一般使用者根本不可能做這樣的操作；它是個bug，但卻是產品賣到市面後幾乎不可能發生的bug。如果很容易就可以改好的話，工程師可以順手將其解掉；但通常這種bug都不會太單純，此時工程師應該陳述修改這個bug可能的風險與副作用，由QC人員評估是否要修改。

 

舉個keyboard驅動程式的實例，一般人工操作不會有反應遲鈍或太靈敏的問題，但有個測試人員使用訊號產生器，模擬每隔1ms壓一次按鍵，發現驅動程式會有漏掉key的狀況，並寫了一張bug sheet。工程師收到這種bug sheet簡直啼笑皆非，就算亞洲第一快手也按不出這樣的速度，即便程式在這樣的狀況下會有問題又如何？既然產品的使用者是一般的地球人，這樣的問題能算是bug嗎？

 

n   只出現一次，無法重現的bug：測試人員作了一系列的操作後發現系統出現異常現象，機器重啟後卻再也找不到重現異常現象的操作手法，而且此後再也沒有人報告發現這個問題。這種現象的爭議很大，我們只能說系統中確實存在著某個問題，但若經過足夠時間與廣度的測試仍然無法重現，RD根本無從找起，它就不該算是一個bug。測試人員最好口頭或書面告知研發主管有這個問題就好，因為沒有重現這個問題的操作手法，RD僅能透過code review試圖找出程式的缺陷，就算找到懷疑的地方，也無法驗明確實已經修正了這個問題，因為本來就無法重現不是嗎？若測試人員把這個問題當作bug給issue出來，將徒增所有人的困擾，RD無法進行debug、測試人員無法驗證新版本是否已經修改、QC人員只能苦苦追問RD問題何在[4]，無論如何也不敢直接將其CLOSE掉。

 

n   REOPEN(某個版本已經改好的bug，在之後的版本又被發現)：REOPEN頻率是外界對程式品質的重要評量指標之一，如果某類別的bug被REOPEN的次數過多，研發主管應主動調查，是否該功能最近有大幅度的修改，或者該功能程式的版本控制出了問題。

 

n   規格修正：客戶可能隨時有新的需求，或想要小幅度修改規格，最簡單的方式就是在Bugzilla上新增一個形態為REQUEST的“bug”；當然這不能算是bug，但在專案後期，如果只是小規模的修改，REQUEST可以採用bug一樣的追蹤方式，同樣有個assigner，相關問題都可以在Bugzilla上討論並留下紀錄。唯一要注意的是製作統計報表時不應該把這個紀錄算在bug數內。

 

n   RD認為修改某個bug可能會造成嚴重的副作用：修改程式可能會帶來副作用(就像台灣厘語 – 抓龜走鱉)，而且越到後期風險越大；如果RD對某個bug有此疑慮，只要它不是主要功能，RD可能傾向於不修改。但其他非技術人員沒有這種sense，他們覺得只要工程師修改程式時稍微謹慎一點，怎麼可能造成副作用？殊不知軟體的複雜度遠超過其所想像。所以RD應該詳述修改這個bug可能的風險，由專案經理會同QC人員決定該如何處置這個bug。

ˊˊ用台　』

 

菜鳥：『我知道上述那些不用修改的bug，其狀態會被設為INVALID或WONTFIX(won’t fix)；而已經改好的bug，其狀態則會從NEW或ASSIGNED，被更改為FIXED；我的問題是：這些bug什麼時候才可以結案(CLOSE)？板子的們題的地方器，他和』

 

測試老鳥：『工程師修改程式後，當然會在自己的電腦上做一個版本，測試無誤後才可以上Bugzilla server宣稱他已經修正了某個bug，並將其狀態改為FIXED。但測試人員只能測試正式版本，所以必須等到下一個正式版本才可能反應工程師的修改。而且並不是工程師說“我改好了！”就真的沒問題了，有時候工程師可能誤解了bug sheet的意義，或者並未完全修正bug sheet內陳述的問題，而最常發生的狀況就是工程師忘了把改過的程式上傳到版本控制server，亦即製作版本時並未用到“最新”的程式，所以測試人員用新版本再測一次，結果無疑仍然是NG。

以下列舉出各個bug狀態的定義：

 

l   新建(NEW)：Bug創建後的初始狀態，尚未確認是否為“合法”的bug，已寄發給該類別bug的預設處理者與其他關係人，但還沒正式指定處理者。

 

l   已分配(ASSIGNED)：也可稱為ACCEPTED，表示這個bug已確認為合法的bug，已經分配給適當的開發人員處理，而且被分配的工程師也接受此bug sheet。

 

l   已修正，待驗證(RESOLVED)：開發部門對bug進行處理或修改後的狀態；RESOLVED的子狀態為：

 

n   FIXED：以找出問題所在，且已經修改。

n   INVALID：開發人員認為這不應該是個bug。

n   WONFIX：這是個不用修改的bug。

n   LATER：開發人員認為Bug在當前版本可以暫不修改，考慮在後續版本中再做修正。

n   REMIND：基本上開發人員不會去修改這個bug，但當有空的時候也許會順手進行修改。

n   DUPLICATE：表示這個bug和某個已經被找到的某個bug是同一個問題。

n   WORKSFORME：是“works for me”的簡寫，意思是說“在我的機器上是OK的啊？”，表示開發人員無法重複這個bug，故無法進行修改。

 

l   重新開啟(REOPENED)：當assigner號稱某個bug已經解掉了，若經過測試人員驗證卻仍然存在，則必須將其狀態改為REOPENED。對於LATER狀態的bug，如果時機成熟，也可將其狀態改為REOPENED狀態。

 

l   已驗證(VERIFIED)：測試部門對修改後的bug進行驗證，若確認結果OK則將其狀態設為VERIFIED。

 

l   未經證實狀態(UNCONFIRMED)：由開發人員自己提交的Bug，是一種初始狀態，待測試人員確定後變為New。

 

l   關閉(CLOSED)：Bug生命週期的結束。對於狀態為VERIFIED的Bug，應由Bug的開啟者，即測試人員主動關閉。開發人員無權關閉Bug。
