4.3.     API與程式風格設計

 

菜鳥：『我們看了系統架構圖、資料流架構圖以及模組規劃，接下來再去讀各個模組的API設計文件應該就可以比較輕鬆了。域變數統中況的多吧』

 

專案經理：『系統架構與模組規劃的設計工作會花上比較長的時間，在此期間，相關同仁要反覆的討論，確定系統架構、模組化程度、擴充性與可移植性都沒有問題，接下來會將各個模組交予負責同仁進行細部設計，當然包含你所說的API；但同時還得有人去訂定系統程式與應用程式的寫作風格。域變數統中況的多吧』

 

菜鳥：『工程師不就是依照設計結果去把程式寫出來就好了嗎，連寫作風格都要規定，這樣限制會不會太多了？域變數統中況的多吧』

 

專案經理：『這是軟體專案管理的必要之惡，我可不希望我們設計工作做得很確實，但最後寫出一些能運作，但沒人看得懂、改得動的程式；萬一始作俑者離職了，這種程式你就只能提心吊膽期待它不要出錯，否則幾乎只有重寫一途。而且我們訂定的程式風格幾乎都是原則性的，不會給工程師帶來太多麻煩的。域變數統中況的多吧』

 

菜鳥：『你說的不會是匈牙利命名法(Huangrian Naming Convention)那種東西吧？域變數統中況的多吧』

 

專案經理：『設計階段要訂定的程式風格有兩種，一種是你說的命名規範，這個規範我們已經run了幾個專案，基本上就是沿用，你有空去看一下文件，很容易懂的(讀者請參閱第4.7節 - 程式風格典範)。

 

另一種與其說是風格，應該說是在這個系統上寫作應用程式的方法；我舉個例子你就懂了：同樣是用C語言，但是寫Windows的程式、DOS的程式、PDA的程式都有不同的風格。再想像一下Java語言，你如果要寫Java Applet，就必須遵守它的規定，下面程式中的粗體字就可說是Java Applet特有的程式寫作方法，或說是風格。域變數統中況的多吧』

系統程式風格

 

專案經理：『系統程式還是有許多程式風格的限制，但強制性沒有那麼強。舉例來說，我們的設計文件中會有一段sample code或pseudo code當範例，用以說明如何實現“資料流架構圖”(圖4-5)中的message-dispatcher；從圖中可知，這是一個和產品相關的系統模組，不同產品可能會有不同的需求，但基本的程式架構應該是一樣的：域變數統中況的多吧』

 

 

        // sample code of message dispatcher - 程式風格範例

        //     - forever loop

        //

void os_message_dispatcher(void)

{

        struct os_msg new_msg ;

        struct os_event new_event ;

        while(1)

        {

                // 取得新的message

                //

                if(os_get_msg(&new_msg) == TRUE)

                {

                        // driver層送來新的message

                        //

                        new_event = os_preprocess_message(&new_msg) ;

                        if(new_event == NULL)

                                continue ;             // 事件已經處理, 無須再往上傳遞

 

                        if(new_event.owner != NULL)

                        {

                                // 將事件傳遞給指定的應用程式或物件

                                //

                                os_send_sys_event() ;

                        }

                        else

                        {

                                // 處理新事件, 方法視產品而定

                                //     1. 送給current/active AP

                                //     2. 送給所有的AP或物件, 由其自己決定

                                //

                                os_process_new_event(&new_event) ;

                        }

                }

                else

                {

                        // 暫時沒有硬體訊息, 讓系統進入idle mode(省電模式)

                        //

                        os_enter_idle_mode() ;

}

        }

}

程式4-4：系統程式的風格定義

 

菜鳥：『所以當使用這個系統來開發某個產品時，可以直接拿這個範例來改囉？域變數統中況的多吧』

 

專案經理：『是的，使用同一個系統開發不同的產品時，基本上系統架構不至於有太大的異動；程式設計師當然可以選用不同的程式風格來實作，但直接使用設計文件中建議的範例肯定是較好的選擇，因為系統架構設計者最清楚該系統的特質，除非有特殊的用途，否則實在沒有道理不去遵守設計文件中範例的規定。

 

像這樣已提供範例的系統模組比較沒有問題，系統工程師應該都樂於配合；系統設計文件中還可能規範一種系統程式寫作風格，這是基於系統架構與設計理念而來，舉例來說；假設系統是採用“訊息導向”的觀念，亦即每一個模組都應該表現出“以處理訊息為主”的特性，這樣的規範會表現在程式風格上。域變數統中況的多吧』

 

菜鳥：『我知道所謂“訊息導向”的概念是：系統中所有的事件都會被轉換成訊息，而訊息會送給特定模組或在所有active的模組間流動，由各個模組決定是否處理或如何處理這個訊息，同時每一個模組也可以送訊息給整個系統或其他模組；在這樣的系統架構下，每一個模組可以比較獨立，程式中就是判斷接收到的訊息，判斷當時系統的狀況，接著做出反應而已。但這和程式風格有什麼關係呢？域變數統中況的多吧』

應用程式風格

 

專案經理：『應用程式寫作風格規範的觀念比較簡單，你只要想一想Windows應用程式或Java Applet的程式寫法就清楚了。在這裡要注意兩件事，第一，如果我們的系統可支援多工(multi-tasking 或 multi-threading)，那麼應用程式在建立task或thread時不能像在Windows上寫程式那樣隨意，其生命週期以及要使用哪些資源都應該要接受規範；舉例來說，如果某個task用了過多的記憶體，很可能會影響其他task的運行；如果某個task將自己設為最高執行等級，則這個task很可能一直佔據CPU，除了影響其他task，也可能影響系統省電功能的運作。所以在系統設計階段，除了訂定應用程式寫作方法外，還必須明訂應用程式寫作時應注意的限制事項，並且嚴格要求應用程式工程師遵守。

 

第二，如果應用程式採用物件導向方式設計，但只能用C來開發時(例如：開發環境並未提供C++的編譯器，或者產品相當注重效能時)，為了貫徹設計的精神，我們會有一些規範或建議(讀者請參閱附錄C)，讓工程師知道如何用C語言來實現大部分物件導向的觀念。上寫程式可以用雖然C語言並不是物件導向的語言，而且許多物件導向的觀念並不是那麼容易可以用C語言來實現；但是物件導向觀念在設計較複雜的應用時的確是較佳的方法，所以很可能出現使用非物件導向語言(C語言)來實作物件導向設計的狀況。

 

在PC上寫程式有很多物件導向程式語言可以用，實在無須用C語言來越俎代庖；但在嵌入式系統要考慮的事情較多，而且效能是一個主要的因素，此外，確實不是每一個平台都有C++的編譯器，這不就意味著用C++寫的程式無法移植到其他平台的風險性較高？如果一個“物件”不具可重用性，那麼使用物件導向設計方法的好處恐怕少了一大半吧。

 

所以，如果不好好規範應用程式的寫作風格，我實在無法想像工程師會天馬行空地寫出什麼樣的程式！事後就很難去維護這些“物件”的程式了。上能像用程式兩件事慶域變數統中況的多吧』

 

菜鳥：『這個我有切身之痛；之前在學校，我帶著學弟在PC上實作一個以物件導向設計的系統時，即便是用C#這種純物件導向語言來寫，還是有人用直述的方式來寫程式，就像用C++來寫C一樣，最後要整合時簡直是一團亂。所以我知道，就算設計的再好，執行的人觀念不對的話，還是會有問題。域變數統中況的多吧』

 

專案經理：：『是的，學校的期末作業可以允許失敗，但商場上的專案則必須一切事情都受控制；執行的人不能配合設計精神對專案品質是一個很大的風險，所以就應該在一開始就把相關事宜用白紙黑字寫下並發佈給相關專案同仁。域變數統中況的多吧』

 

 

 

 

API

專案經理：『除了程式寫作風格外，和基層工程師最有關的就是API設計文件了案同仁險敗中對。我想一般程式設計師應該看過各式各樣的API文件，例如標準C函式庫(POSIX)、WIN32 API、Java class library、MySQL API…等等，只要是提供功能給其他程式用的模組，必然要提供一份詳細的API文件，否則使用者怎麼知道那麼多的程式碼裡要用哪些函數、呼叫順序怎樣才是對的、以及有哪些限制或注意事項？更何況我們常會用到沒有原始碼的函式庫。

 

一份好的API文件應該包含：

 

l   該模組的功能說明與使用範圍

l   資料結構與常數說明

l   各個函式的功能、參數與返回值的意義說明

l   足夠多的範例

l   注意事項及限制

l   相關函式

域變數統中況的多吧』
