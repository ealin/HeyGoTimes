8.2.     Stack

 

系統老鳥：『不同的專案會有不同的記憶體配置，例如我用過一顆CPU，它在CPU內部有一小段boot程式，可以從NAND Flash載入程式到RAM中，所以這種CPU就可以不需要由ROM或NOR Flash來啟動程式，系統可以只有NAND Flash與SDRAM這兩種“便宜又大碗”的記憶體；這種配置就和前一節那個例子大異其趣碗式憶體配置。但無論是什麼樣的應用，有些記憶區段都是必要的：

 

a.          程式段(text段與rodata段)，可直接在ROM或Flash中執行，也可將某個模組搬移到速度較快的RAM裡執行。

b.         有初始值的全域變數(data段)，會佔據執行檔空間，執行時期必須將其從ROM或Flash搬移到RAM。

c.          沒有初始值的全域變數(bss段)，不會佔據執行檔空間，執行時期必須將該區段的內容全部設為0。

d.         Stack(堆疊)

 

其中a、b、c三項內的程式或變數，其位址都由linker產生(當然必須由系統工程師在link script中描述)，我們在講開發環境時都已經說過了，在此就不再贅述。在此我們來談談stack，你知道stack有哪些用途嗎？ˊˊ用台　』

 

 

Stack的用途

 

菜鳥：『由CPU內部都有一個SP(stack pointer)暫存器看來，可以得知stack 是CPU運作不可或缺的機制，從CPU的角度來看，stack的用途是：

 

l   在執行“call”指令時，會將返回位址儲存(push)到目前SP暫存器指到的位置(SP始終指到stack的頂端)。

 

l   當執行“ret”(return)指令時，會自目前stack的頂端取回(pop)返回位址。

 

l   當中斷產生時，中斷控制器會將發生中斷的位址(返回位址)以及PSR(狀態暫存器)儲存(push)到stack裡。

 

l   當ISR執行“iret”(interrupt return)指令時，CPU會從目前stack頂端pop出PSR與被中斷的位址。

 

l   我們都知道stack這個資料結構特性就是先進後出，CPU提供“push”與“pop”指令供程式設計師操作stack，最常見的用途就是暫時將某些儲存暫存器的值存在stack中，在某些動作後再“依序”取回這些暫存器的值，如下面的例子。此外“push”與“pop”的動作必須是對稱的，push多少東西到stack，就必須記得依序pop多少東西出來，否則下次存取stack時就資料就會亂掉。

 

// 將暫存器r0~r2的值依序儲存到stack中

//

asm("push r0") ;

asm("push r1") ;

asm("push r2") ;

...

...

// 自stack中取回儲存的值

// 必須注意的是, 最後面push的暫存器, 要最早pop出來

// 此即stack所謂後進先出的特性

//

asm("pop r2") ;

asm("pop r1") ;

asm("pop r0") ;

 

 

如果從C語言程式的角度來看，stack memory的用途更多了：

 

l   Local變數會儲存在stack中，如果應用程式工程師不知道可以用的stack有多大，一不小心就會把stack用爆了。此外要注意的是：用於嵌入式系統的編譯器通常不會產生“將區域變數的值設為0”的code，也就是說，如果程式設計師沒有明確指定區域變數初值的話，區域變數的初值可能是任何的值。如果程式設計師疏忽將沒給初值的區域變數拿來使用，則結果自然是無法預測的，例如：

 

int test_fun(void)

{

 // 這是一個極端無厘頭的程式, 在PC上應該會得到的傳回值是0

 // 但在嵌入式系統中, 則可能傳回任何值,

 // 端視stack memory當時的狀態

 //

 int a ;

 int b ;

 

 b = a ;

 return b ;

}

 

l   函數呼叫時返回位址當然會依照目前SP暫存器，儲存在stack中，這是CPU的機制，C語言的函數自然也是如此運作。

 

l   函數呼叫時參數的傳遞可以透過暫存器，也可以透過stack，端視編譯器的策略而定；通常如果參數個數不多的話，編譯器會傾向用暫存器來傳遞參數，反之則把參數存在stack中。

 

所以如果函數呼叫太多層，每一層函數的返回位址、參數、區域變數都會一直吃掉stack，例如遞迴的結束條件沒控制好，就可能發生stack overflow的現象，結果當然是其他的記憶體被壓到與破壞；雖然我沒有實際碰過這樣的bug，但你之前說過，stack overflow的問題很棘手，因為現象不會在押到的哪一瞬間反映出來，會一直等到被破壞的記憶體被使用時才會知道出了問題，而被記憶體被破壞的程式可能和造成stack overflow的程式一點關係都沒有。

 

以下是個遞迴的例子，函數內定義了一個100 bytes的陣列，100 bytes其實也不算是太大；但當遞迴開始返回時，stack用量已經累加了至少20000 bytes，一般嵌入式系統不太可能配置這麼大的stack，亦即這個函數執行完後，必然已經破壞了其他的記憶體。ˊˊ用台　』

 

 

void main(void)

{

 recursive_fun(0) ;

}

 

 

void recursive_fun(int counter)

{

 // 這個函數每被遞迴一次, stack用量至少會累加100 bytes

 //

 char A[100] ;

 

 if(counter <= 200)            // 遞迴結束條件

 {

        // 當遞迴結束時, stack用量至少已經累加了100*200=20000 bytes

        //

        return ;

 }

 else

 {

        recursive_fun(counter+1) ;

 }

}

Stack Overflow

 

系統老鳥：『原本一般的工程師也不需要知道這些細節，因為在PC有虛擬記憶體，作業系統與編譯器可以分配一大段的位址空間當作stack，所以程式設計師確實不太需要擔心stack overflow這種事情。但開發嵌入式系統的工程師就不能不知道stack運作的概念，否則程式寫作時完全不顧stack的大小，區域變數亂用，最後發生的問題將會被歸類為“系統不穩定”，你也知道這種bug是最難找的。

 

測試人員只知道某個功能(以下稱之為功能A)單獨執行時OK，可是有時候做了一系列操作後，再去執行功能A就會出問題，這種時好時壞的問題最容易造成管理階層與QA/QC同仁的緊張，偏偏出問題的程式(功能A)怎麼看都看不出問題，只知道它會用到的某個變數或某塊記憶體的值被破壞了。有經驗的工程師會推測應該是stack overflow或指標誤用的問題，但系統裡的函數那麼多，我們在debug時怎麼知道到底是哪一個函數在什麼時候造成stack overflow，並且破壞了功能A要用的記憶體呢？碰到這種問題時，測試人員要盡量歸納出一定會NG的操作手法，工程師才能循著這個手法的路徑上找出有問題的函數[1]。ˊˊ用台　』

 

菜鳥：『總之不要用太大的區域變數就不會有問題了吧？ˊˊ用台　』

 

系統老鳥：『區域變數的大小當然要有所節制，假設系統的stack只有2K Bytes，工程師在某函數內定義了一個區域變數“char local_var[2050]”，如果不去操作可能還沒事，如果作了諸如“local_var[2049] = 0 ;”的設值操作，當場就超過了原先分配給stack memory的記憶體配置，至於被壓到的記憶體是否有意義，屬於什麼用途則視系統整體的記憶體配置而定。根據我的經驗，嵌入式系統開發案的記憶體通常都不會有剩，也就是說RAM裡面總是塞得滿滿的，甚至當記憶體不夠時，系統工程師馬上就會想到可以從stack memory中“偷”一點來用於其他用途。

 

除了區域變數的大小必須節制之外，呼叫遞迴函數的層數也要小心，每一層會用去的stack空間包含這個函數的返回位址、參數與所有區域變數的size總和，如果不稍作計算的話，多呼叫幾層後同樣會造成stack overflow。

 

避免用爆stack是每一個工程師的責任，除此之外，區域變數的操作也要謹慎，我們來看看以下的程式，先不管程式的用途，你仔細看看，是否能察覺這個程式有什麼問題嗎？ˊˊ用台　』

 

void test_fun(void)

{

 char a[10] ;

 

 // 指標b指到陣列a頂端的下一個byte

 //

 char *b = a + 10 ;

 

 // 將陣列a的每個元素設為100

 //

 do

 {

        b-- ;

        b[0] = 100 ;

 } while(b >= a) ;

}

 

菜鳥：『這個函數只是要把陣列a的每一個元素都設為100，如果說有什麼問題的話，就是用了太複雜的方法來處理這麼簡單的事情。ˊˊ用台　』

 

系統老鳥：『乍看之下是沒什麼問題，但這個函數只要一執行很可能就會當機。它有個簡單的邏輯錯誤，當b一路遞減到a時，因為while迴圈的條件是b>=a，所以最後一次進入迴圈時b會等於(a-1)，你認為把陣列a開頭的上一個元素設為100會發生什麼事？ˊˊ用台　』

 

菜鳥：『這叫做stack underflow吧，和overflow不一樣嗎？前者是破壞stack內的記憶體，後者是破壞stack外的記憶體，總之就是記憶體被破壞了；這是程式的bug，應該很容易可以查出來吧。ˊˊ用台　』

 

系統老鳥：『說穿了當然很容易，剛剛你不就覺得這個程式沒什麼問題嗎？這種記憶體邊界的問題在使用某些複雜的演算法時還真的很容易發生，尤其是用一個索引變數來存取陣列時，不謹慎很容易就會overflow或underflow，如果是讀取的動作，那麼發生的錯誤是立即性的，通常比較容易查；如果是寫的話，問題會拖到被破壞的記憶體被用到時才爆發出來，這種問題就比較難查了。

 

我們來分析剛剛那個例子，陣列a會被定址在stack內，而a是這個函數的第一個區域變數，這個函數剛執行時stack的狀態如下圖，你現在知道剛剛那個程式的小疏忽會破壞什麼了沒？ˊˊ用台　』

Stack & RTOS

 

系統老鳥：『最後我們從RTOS的角度來看stack memory的用途；一般我們用於嵌入式系統的CPU都不會有虛擬記憶體的功能，也就是說系統無法像Windows或Linux一樣為每一個程式配置獨立的虛擬位址空間；在這樣的硬體限制下，你知道RTOS是如何實現多工的功能嗎？ˊˊ用台　』

 

菜鳥：『我看過Windows的技術文件與μCOS II那本書，對作業系統的多功多少還有點了解；在Windows或Linux上的多工應該可以分為兩種，一個是Multiple-Process，另一個則是Multiple-Thread；每一個process有自己獨立的位址空間，而且每一個process內可以建立多個thread，所以process內的所有thread則位於同一個位址空間。

 

舉個簡單的例子來說明位址空間的概念，可以把process想像成不同的執行檔(.EXE)，因為不同的process有獨立的位址空間，假設不同的process都去存取同樣的位址(如0x100000)，實際上作業系統會將其mapping到不同的實體位址，所以絕對不會互相衝突。至於同一個process內的不同thread，因為共用位址空間，所以不同thread存取同樣的位址時就可能會引發衝突[2]。因為Thread沒有自己的位址空間，所以thread間切換的複雜度比process間的切換低很多，而且因為多個thread共用位址空間，所以thread間的通訊相對單純；總之只要做好critical section保護，thread的效能較好、程式寫作較簡單，卻同樣可以達到多工的效果，這也是目前multiple-threading的程式設計方法蔚為風潮的主要原因。

 

用於嵌入式系統的RTOS的多工功能就是上述的multiple-thread，通常嵌入式作業系統的書或RTOS的網站會把這個功能稱為multiple-task，其實是一樣的東西，只是名詞不同而已。總之就是所有的執行單位，不論稱之為thread或task，都共用同一個位址空間。RTOS多工功能的實現方法我不是很清楚，但我只知道每一個執行單位都要有自己的stack，一般嵌入式系統多工程式的風格如下：ˊˊ用台　』

 

// 宣告main task與idle task的entry point

//

void main_task (void *pdata)

void idle_task (void *pdata)

 

 

// 定義main task與idle task的stack空間

//

OS_STK *main_TaskStk[1024];

OS_STK *idle_TaskStk[512];

 

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

// main function

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

void main (void)

{

 INT8U err;

 

 OSInit();      // 初始化task管理系統

 

 // 建立main task

 //

 err = OSTaskCreate(

        main_TaskStk,              // main task的entry point

        (void *)0,                        // 傳給task的參數

        7,                             // 優先等級(最高)

        &TaskStk1[1023],       // stack由高位址往低位址長

        1024,                      // stack size

        );

 

 // 建立idle task

 //

 err = OSTaskCreate(

        idle_TaskStk,         // idle task的entry point

        (void *)0,                        // 傳給task的參數

        0,                             // 優先等級(最低)

        &TaskStk[511],    // stack由高位址往低位址長

        512,                                // stack size

        );

 

 // Start Multitasking

 // 啟動task管理系統, 系統會找出最高優先等級的task來執行

 //

 OSStart();

 

 // <-- Multitasking啟動後, CPU執行全會在task之間輪替

 //   所以OSStart()並不會return回到這裡!

}

// main task

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

 ...

 {

        ...

 }

}

系統老鳥：『不錯，你的觀念很正確，很難得有菜鳥可以把這些原理講得頭頭是道。從上面這個範例可以看出來，其實所謂的thread或task不過就是一個函數罷了，編譯器根本就沒有thread的觀念；舉個極端的例子，就算在task2中呼叫task1()，即使邏輯大有問題，但因為語法完全符合C語言的規範，所以編譯器絕對不會報錯。亦即thread之間完全沒有隔閡，當thread交互執行時，如果系統還是只有一個stack肯定會天下大亂，所以RTOS的多工功能會讓每一個thread有自己的stack；Thread的stack和之前提到之系統stack的功能一樣，包含函數呼叫時的參數、返回位址，以及區域變數等，只是每一個thread可以依據需求配置不同大小的stack。

 

你可以思考一下，剛剛我們提到許多程式設計師的注意事項，用以避免破壞系統的stack；如今每一個thread有自己的stack，而且還是用global array的方式配置記憶體，寫程式時是不是應該更小心一點？』

 

菜鳥：『在沒有多工需求的應用時，系統工程師會依據記憶體的使用狀況，分配一塊記憶體區域當作系統的stack，stack的起始位址與結束位址是由系統工程師指定，而非連結時期由Linker算出來的。如果我是系統工程師，我想我在指定stack的區域時會在前後保留一點緩衝的空間，如果有不可預測的狀態導致stack overflow了一點，也不至於讓系統當機。

 

如果是用global array的方式配置每一個thread的stack時，對linker而言，這個array和一般的全域變數並沒什麼不同，所以當然會和其他全域變數定址在一起(bss段)，它前後各是什麼變數是由linker決定的，也就是說，當stack overflow或underflow時一定會破壞到別的變數，至於怎麼查出哪些變數會被破壞，我就不知道了。ˊˊ用台　』

 

系統老鳥：『你忘了Linker會產生Symbol Table或MAP檔嗎？在裡面你可以很容易查出變數的前後關係，想知道stack操作可能破壞的變數，在MAP 檔中找出代表stack空間的全域變數，再看看它的前後各是哪些變數就知道了。ˊˊ用台　』

 

Context Switch

菜鳥：『從前在修作業系統時有學過context switch，記得當時老師也只是隨便帶過，反正就是作業系統執行程式間的切換，至於實際是怎麼實現的，實在一點概念都沒有。ˊˊ用台　』

 

系統老鳥：『基本上可以想像是多個thread輪流在CPU上執行，但無論如何CPU只有一個，一段時間中只能有一個thread獲准取得CPU的使用權，同時RTOS必須妥善保存其他被擱置的thread的執行狀態，則當換另一個thread執行時，它才可以順利地從上次被task-switch的位置繼續執行。RTOS用一個資料結構紀錄各個thread的執行狀態，我們稱這個狀態為context(所以task間的切換也稱為context-switch)，每一個thread的context中包含了該thread上次被切換出去時所有暫存器的值以及一些系統資訊；而thread的context會儲存在自己的stack中，context switch也是透過操作stack完成。ˊˊ用台　』

 

 

菜鳥：『不太了解context switch和stack有什麼關係，你可以說明一下context switch是怎麼完成的嗎？ˊˊ用台　』

 

系統老鳥：『因為這不是我們的主題，我只能大概講一下概念，如果你想知道更多的細節，有空好好K一下μCOS II那本書，裡面有很清楚的範例。Context Switch的原理很簡單，但程式並不是很好懂；你要先知道一件事，一般狀態下，CPU是循序地在執行程式，程式可透過某些手段改變PC(program counter)的值，就可以迫使CPU跳到另一個位址去執行，一般的方法是function call與goto，另外有個比較技巧性的方法是：修改在stack中的函數的返回位址，則程式執行return時就不會返回原來的caller function，而是跳到另一個位址去執行。請參考下圖及其中的pseudo code：ˊˊ用台　』

 

圖8-3：透過修改stack的技巧改變執行順序

 

菜鳥：『因為return命令會到stack中抓取返回位址，原本B函數執行完畢後應該要返回A函數，但經過B函數對stack的“惡搞”後，結果B函數返回時居然跑到C函數去了。這個技巧確實很神奇，但和context switch有什麼關係？ˊˊ用台　』

 

系統老鳥：『如果你能接受這個技巧，接下來我們才能繼續說context switch的原理。你想想，剛剛那個例子的流程像不像task A執行了context switch的函數(B())，然後就跳到task C去執行了？如果我們有辦法知道A函數中“ a = 0 ;”這行指令的位址(被task switch的位址)，我們是不是也有辦法從task C跳回task A？如果函數B有一個表格紀錄所有task的位址，那麼它是不是有辦法切換到任意的task？』

 

菜鳥：『對耶~真神奇！一般的function call每次都要從函數的開頭執行，利用操作stack的技巧就可以做到類似task切換的效果。ˊˊ用台　』

 

系統老鳥：『真正context switch的作法比上述的例子更具技巧性，它不但操作stack的內容，而且因為每一個task都有自己的stack，所以還必須改變SP(stack pointer)暫存器的值，這個已經是我們系統中操作stack技巧的極致了。Context switch通常是透過一個可重入(reentry)的函數完成，每一個task被切換出去的位址正是這個函數的最後一行(return)，有點玄對不對？看了以下的pseudo code可能會清楚一點：ˊˊ用台　』

 

void context_switch(void)

{

 // 因為context switch會操作Stack, 為減少複雜度, 最好不要用區域變數

 

 // 執行context switch時最好把中斷關掉，避免執行ISR時造成混亂

 //

 drv_disable_interrupt() ;

 

 // 接下來要開始操作stack與SP register，所以從現在起就不會再呼叫函數了

 

 // STEP1: 儲存目前task(TASK-A)的所有暫存器到stack中

 //

 asm("push r0") ;

 asm("push r1") ;

 ...

 ...

 asm("push r15") ;

 

 // STEP2: 儲存目前SP暫存器的值到global變數 – “_CurrentThraedContext”中

 //       等TASK-A要被切換回來時, 可以從這個變數中取回TASK-A的stack位址

 //       (重要: stack中的內容包含所有暫存器的值以及最重要的“return address”)

 //

 asm("ld _CurrentThraedContext, sp") ; // _CurrentThraed = SP

 

 // STEP3: RTOS的排程器會選定下一個要執行的task(TASK-B)

 //       並將其被切換出去時所儲存的SP值取出(就是TASK-B執行step 2的結果)

 //       置於global變數“_NextThraedContext”中

 //       此步驟將SP暫存器的值設為“_NextThraedContext”

 //       這個步驟執行過後, SP已經指到另一個task(TASK-B)的stack

 //       還記得stack裡有什麼嗎 - 所有暫存器的值以及最重要的“return address”

 //

 asm("ld sp, _NextThraedContext") ;      // SP = _NextThraedContext

 

 // STEP4: 開始將CPU狀態恢復成另一個task被切換出去時的狀態

 //

 asm("pop r15") ;

 asm("pop r14") ;

 ...

 ...

 asm("pop r0") ;

 

 // 把中斷打開

 //

 drv_enable_interrupt() ;

 

 // 目前stack頂端的內容為何 – 正是RETURN ADDRESS !

 // 執行return後, TASK B就會繼續執行

 //

 return ;

}
