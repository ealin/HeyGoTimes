11.4.            Tuning

 

菜鳥：『測試人員已經開出了幾張系統效能不佳的bug單，我們的CPU速度這麼慢，記憶體size又小，效能不好是可預期的吧，我想同樣的程式在PC上一定跑的嚇嚇叫！這種效能的問題要怎麼解啊？ˊˊ用台　』

 

系統老鳥：『效能問題可說是嵌入式系統開發者心中的痛，除了硬體效能本來就不能與PC相提並論外，使用者對電子產品效能的要求幾近嚴苛；使用者可以忍耐Windows上的漏斗出現，卻不能接受洗衣機按下洗衣鈕後很久後才反應，更不要說其他有real-time需求的應用了，晚來的答案等於錯誤的答案！如果ABS煞車系統要一兩秒才能算出正確的控制方式，這種車子誰敢開？

 

有關嵌入式系統效能最佳化的學問恐怕可以寫一本書了；不同的產品有不同硬體設備、不同的效能需求、以及不同的複雜度，所以效能調整的困難度也是天差地遠。我會講一些原則性的觀念，如果你計算機概論、演算法、資料結構、作業系統的課有認真上，老師講的話你都有在聽的話，碰上效能問題時，應該都能想出對應的方法來作試驗。但能不能真的把效能調整到符合需求，則端視硬體與系統架構的極限，要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

菜鳥：『你講得很輕鬆，我還是一頭霧水，可不可以講得實際一點，碰到效能問題該怎麼著手？有沒有可遵循的步驟？ˊˊ用台　』

 

系統老鳥：『雖說最佳化是case by case的事情，但還是有幾個重點：

 

l   在進行程式最佳化之前，先要有一個穩定、可靠的版本：而且在確定最佳化未完成前，不要將程式匯入正式的版本之內。

 

l   調整編譯器最佳化的等級；但務必記得一件事，編譯器無法幫程式設計師完成所有最佳化的任務，尤其是程式風格不佳，或寫得零零落落的程式。

 

l   優先使用改良演算法的手段來做效能調整；舉例來說，無論再怎麼精簡程式，也無法彌補bubble sort與quick sort間的效能差距。

 

l   以空間換取時間；如果記憶體還有剩的話，可以試著思考利用多餘的記憶體空間來改良演算法，進而改善效能。

 

l   不要把程式最佳化與“用組合語言修改”混為一談；首先應該找到程式的熱區(hot spot)，也就是程式中效能的瓶頸所在，再利用所有你在學校學到與最佳化有關的技巧來試著修改它，例如：

 

n   盡量改寫程式中用到乘法、除法與餘數計算的部份，例如乘以2N(例如a = b * 8 ;)可以改成左移N個bit(a = (b << 3) ;)：除以2N(例如a = b / 8 ;)，可以改成右移N個bit(a = (b >> 3) ;)。

 

n   最大限度地減少分支(if…else if…else或switch指令)的數目與複雜度。

 

n   盡量循序處理資料；例如陣列a[100][200]，要將陣列內每一個元素都設為0x0A，方法一的效能會比方法二的效能好：

 

int i, j ;

char a[100][200] ;

 

// 方法一

//

for(i=0;i<100;i++)

        for(j=0;j<200;j++)

                a[i][j] = 0x0A ;

 

// 方法二

//

for(i=0;i<200;i++)

        for(j=0;j<100;j++)

                a[j][i] = 0x0A ;

 

圖11-8：盡量循序處理資料

 

 

n   檢查並盡量減少for迴圈或while迴圈內的迭代次數。

 

n   在迭代次數較多的for迴圈或while迴圈，盡量精簡迴圈內的程式。下面這個例子中，迴圈裡計算字串長度的程式是多餘的，只要在進入迴圈前做一次即可。

 

const char string[] = "this is an example" ;

int length ;

 

for(i=0;i<99999;i++)

{

 // 計算字串的長度

 //

 length = strlen(string) ;

 

 do_something(length) ;

 

}

 

n   牢記不同的記憶體有不同的速度，盡量把熱區(hot spot)的程式或資料搬移到速度較快的記憶體上執行或操作。

 

l   對大部分CPU而言，一次操作偶數或四的倍數bytes的資料會有比較好的效能，檢查程式中是否可能用到奇數的位址，或一次操作奇數個bytes的程式碼；如果有的話，應參考CPU data sheet試著修改它。以下的程式要將陣列“buffer”內的每一個byte都設為0x0A，方法二的效能會比方法一的效能快上數倍。

 

int i ;

char buffer[10000] ;

int * pointer ;

 

// 方法一

//

for(i=0;i<10000;i++)

        buffer[i] = 0x0A ;

 

// 方法二

//

pointer = buffer ;

for(i=0;i<(10000/4)l;i++)

        pointer[i] = 0x0A0A0A0A ;

 

 

l   不要試圖從頭到尾改寫某個模組為組合語言，先檢查編譯器產生的組合語言碼是否有明顯的效能問題，再根據這份組合語言碼來修改即可。在修改時，應仔細研究CPU各個指令的效能，盡量使用效能較高的指令。

 

l   去除不必要程式碼；許多用以除錯的程式碼會忘了去除，導致在正式版本中，系統去執行了許多沒有必要的程式段。

 

最佳化的觀念與技巧差不多就是這些了，我還是那句話，如果你程式設計與計算機系統的觀念夠紮實的話，等你碰到問題時你應該就會知道有哪些方法與技巧可以使用，不用太擔心。要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

執行時間量測

菜鳥：『剛剛提到在著手調整系統效能之前，應該要先找出程式效能的熱區(hot spot)，我想知道所謂的熱區是怎麼找出來的？ˊˊ用台　』

 

系統老鳥：『我們說某個功能或模組的效能不佳，其實就是指相關程式完成該功能所需要的時間太長或無法合乎需求；為了找出程式中效能的瓶頸所在，我們需要能夠“測量CPU執行某段code所需要的時間”的工具或方法。要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

菜鳥：『應該用一個timer來計時就可以了，在欲測量執行時間的程式段開頭啟動timer，結束後即刻停止timer；timer ISR內有一個counter，timer ISR每被執行一次，counter就累加1。最後由counter的值乘以timeout的週期，即可算出某個程式段的執行時間。ˊˊ用台　』

 

系統老鳥：『用timer是個不錯的方法，但要注意可能的誤差；除非待測程式段的執行時間較長，例如一秒或1ms以上，否則我並不建議用這個方式。首先timer有其精度，我們不能把精度設太細，否則待測程式執行時，timer中斷會發生的很頻繁，我們量測出來的時間段中，有一部分是在執行timer ISR。此外，我們也不能把timer精度設太粗，否則量測出來的數據無法表達細微的改變，這對效能調整的工作毫無幫助。

 

其實有個方法很實用也很簡單，如果你把待測程式段執行個十萬或一百萬次[1]，就可以用碼表量測，量得的時間再除以十萬或一百萬即可得到執行一次的時間。要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

菜鳥：『用碼表按，這樣誤差不是很大嗎？ˊˊ用台　』

 

系統老鳥：『教科書當然不會教你這種似乎不是很專業方法，但這的確是相當實用簡單的手段，而且你說的誤差在除以100萬之後幾乎可以忽略了，實務上我們就是用這種方式來量測程式段的執行時間。。要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

找出熱區

系統老鳥：『有了量測執行時間的工具，接下來我們要試著找出程式中對效能影響最大的熱區；說說看你所能想到的方法？要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

系統老鳥：『如果程式是自己寫的，應該很清楚效率不佳的程式區段在哪裡吧；不然你剛剛也提到了幾個重點，應該都是判斷熱區所在的線索：

 

a.      迭代次數很多的迴圈

b.     乘除法運算頻繁的程式區段

c.      頻繁被呼叫的函數

d.     遞迴的程式段

e.      頻繁task switch的程式段

f.        容易造成heap碎塊的程式

g.     …

 

因為在PC上執行什麼都很快，所以我們只能在模擬器上trace程式的邏輯，得知哪些程式段頻繁地被執行，推測可能造成效能瓶頸的程式區段；至於要觀察修改過的程式效能有沒有改善，還是得到實體機器上測試。有必要的話，就應該實際量測修改前和修改後，“熱區”程式段執行時間的改變；或者直接在實體機器上trace程式，當執行到效能不彰的功能時，整個系統好像沒在動且不知道正在做什麼運算時，利用debug環境的暫停功能讓程式暫時停止執行，再繼續逐行trace，應該就能夠知道到底程式在哪一個區段卡了那麼長的時間。要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

系統老鳥：『如果負責執行效能調整的工程師，對你上面說的這些道理有概念，自然可找出效能的瓶頸；就怕這個工程師本來就不認為自己寫的程式有效率上的問題，讓他再看N次，他還是覺得這已經是最佳的寫法了，沒有改善的空間。為避免這種類似裁判兼球員的問題，研發主管會強迫執行下列兩個動作：

 

a.      Profiling：統計可疑函數或程式區段執行時期的資訊，所謂的“資訊”指當使用者執行了某個功能時，期間相關的函數各被執行了多少次、被執行前後memory pool或stack的狀況、每次被執行各用了多少時間…等。

 

如果實際上某個模組的效能確實不佳，但負責的工程師卻始終找不出效能低落的原因時，我們會要求工程師使用profiling的方法產生執行時期資訊的書面報告，再由相關同仁會診，一齊找出所有值得懷疑的地方。然後依此結果，縮小範圍，這次以函式內的程式區段為單位，再做一次profiling。此來不但可以讓其他同仁參與效能問題的分析與討論，而且依據這些資訊應該可以輕易地找出“最值得”花心力去修改的程式段。

 

b.     Code Review：工程師各有專長，對某些領域的觀念難免會有偏頗之處；對已造成效能瓶頸的程式模組，我們不能只讓原先負責的工程師去檢查，對自己寫的程式難免會陷入迷思，總是會有所偏執。最好的方法是召開code review會議，由負責的工程師主持，講解該模組的架構以及程式細節，並盡量要求有經驗的工程師參加。

 

這種會議與之前檢查工程師的程式是否依照設計規格、是否符合規定的程式風格…等的code review不同；這樣的會議應該是一種腦力激盪，所有參與同仁都應該試著找出程式中效能或架構的缺陷，或者提出任何可改善該程式效能的方案。

要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

菜鳥：『可以想見這種code review會議一定是高手過招，有機會可不可讓我也參與一次。ˊˊ用台　』

 

系統老鳥：『其實也沒你想的那麼神奇，除了改用效能較好的演算法外，最佳化的技巧其實都是很基本的觀念而已；剛剛不是說工程師常對自己寫的程式有所偏執，容易陷入迷思；有時候問題真的很簡單，但自己就是看不出來，這就是所謂的當局者迷吧。要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

在較快的記憶體中執行程式或操作資料

系統老鳥：『除了上述最佳化的技巧之外，在嵌入式系統中有個很有效的方法，就是將造成系統效能瓶頸的程式區段，搬移到速度較快的記憶體中，例如程式在ROM裡執行肯定比在RAM裡執行慢，在外部RAM裡執行一定比在CPU內部的RAM裡執行慢，這是物理性的限制。有時候演算法調了半天沒有效果，結果只要把熱點的程式段搬移到速度較快的記憶體執行，效能就可以提升約50%以上。

 

我遇過一個最極端的例子，是有關語音壓縮演算法的模組，其中要用到許多複雜的數學運算，首先我們先從學術論文中找到已被證明為最佳的演算法，發現效能在我們的CPU上還是不行；接著我們把程式從ROM移到RAM裡執行，然後再把程式從外部RAM移到CPU內部的RAM裡執行，做了這些步驟後，聲音品質只能說是差強人意，還需要進一步的調整。一直到了這種山窮水盡的地步，我們才決定用組合語言改寫相關程式，最後效能剛好可以接受；對所有參與這次調整的同仁，能在這樣慢速的CPU進行調速都是相當難得的經驗，幾乎什麼技巧都用上了。要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

菜鳥：『除了程式之外，是否“資料”也可以比照辦理？有時候可能程式的演算法沒有大問題，改善的空間不大，但其所操作的資料是儲存於較慢的記憶體中，所以導致整體效能就是快不起來。就像同樣的演算法，直接操作硬碟中的資料，與“先將資料載入RAM再操作”的效能一定會有所差異。ˊˊ用台　』

 

系統老鳥：『沒錯，其實道理是一樣的，如果某個功能的瓶頸乃在於操作資料的程式，最佳化這個程式的方法之一就是在較快的記憶統中操作這些資料。廣義來說，各種資料儲存媒介的速度由慢到快依序是：

 

1.      必須經檔案系統存取的NAND Flash (最慢)

2.      直接存取NAND Flash

3.      必須經檔案系統存取的NOR Flash 或Mask-ROM

4.      直接存取NOR Flash 或Mask-ROM

5.      CPU外部的RAM

6.      CPU內部的RAM

7.      CPU的一般暫存器 (最快)

 

需要注意的是：速度越快的記憶設備通常越貴，而越貴的記憶體，在系統中可用的size自然也比較少；所以在改善效能時，勢必要把造成效能瓶頸的熱區侷限在更小的範圍內，才有辦法將其塞進速度較快的記憶體中。要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 

菜鳥：『從前在PC上寫程式幾乎不用考慮到效能的問題，就算完全不考慮演算法，或者記憶體無限制的亂用，程式還是可以執行的很順暢；我想我以前那些程式如果可以拿到我們現在的機器上執行，一定慢的和烏龜爬一樣。現在我寫我們專案的程式時，就算只是和硬體無關的應用程式，我都會小心翼翼地考慮到我們之前說到的注意事項，例如記憶體要謹慎使用、使用有效率的演算法…等等；我想這是我進入嵌入式系統這個領域以來最大的改變與收穫。識系統的程式時定慢的的機器上執行果拿到問題區段卡了那麼長的時間，那ˊˊ用台　』

 

系統老鳥：『一旦你寫程式的心態與習慣改變後，就算以後不是開發嵌入式系統，你都能寫出高效能、高品質的程式。要原因之ㄧ的學到了了的地方種問題應該是程式設計師的基本技能吧，ˊˊ用台　』

 


[1] 執行的次數依待測程式段的約略執行時間而定；實務上，總共執行時間約為幾十秒或分鐘等級即可。
