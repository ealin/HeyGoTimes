5. 建構嵌入式系統開發環境

 

專案經理：『我們系統第一版的開發環境已經建構好了，麻煩你到server上把開發環境及其文件與目前版本的程式抓到你的電腦，試試看能不能做出執行檔，然後下載到機器上看能否執行。下午系統組的主管，他在這個專案負責系統整合(以下稱其為系統老鳥)，要向所有工程師報告如何使用這個開發環境以及相關注意事項；在此之前，我們應該先使用一下，看看有沒有需要改善的地方。責人行程式境』

 

菜鳥：『我前幾天已經把整個開發環境抓下來了，設計文件中有一份“開發環境使用者手冊”，我按照裡面的步驟試了一下；但是當時版本的程式好像有點問題，有些程式檔會編譯不過，當時我向系統組的主管報告了這個狀況，後來我就沒有再試了；我想現在可能改好了，等下我就會再去試試看。

 

可是我還是有些事情不了解，我知道開發環境的最終目的就是要產生執行檔，但裡面除了廠商提供的編譯器等工具外，似乎還包含許多我們自己寫的，像程式又不是程式的檔案(makefile & link script)，而這些檔案的內容似乎不是那麼直覺；此外還有許多我們自己寫的工具，我想知道所謂的開發環境到底包含什麼？』

 

專案經理：『就像我們要建構系統平台的觀念一樣，每一個電子產品開發專案用的CPU或產品特性可能都不一樣，所以編譯工具、製作出可下載到機器之執行檔的流程可能都大異其趣；我們希望大部分的工程師只要專注在自己負責的開發工作，不要再花時間去適應各個不同的開發工具或執行檔製作流程；我們會有專門的同仁把這些繁複的動作包裝成簡單的步驟。

 

具體來說，應用程式工程師只要在整合開發環境(IDE)上選擇“build project”或在DOS視窗上執行build.bat這個批次檔，如果程式無誤的話，應該就能製作出可以下載到機器上運行的執行檔，最好一切都是自動的。所有做出執行檔的動作與細節都包裝在build.bat內，這些流程由負責系統整合的同仁統一控管，應用程式工程師其實不用理解太多，只要會用即可。責人行程式境』

 

菜鳥：『之前我們在Visual Studio上開發PC程式時，不論是一般執行檔、DLL、ASP等，只要使用Visual Studio的功能就可以把軟體專案管得好好的，從來也沒想過要還要去建構開發環境。我也在Linux上寫過程式，以前要自己寫makefile，但是目前已經有諸如glade、designer、kdeveloper等開發環境，寫Linux的應用程式似乎也不是什麼困難的事情。所以其實我還蠻想要知道嵌入式系統的開發環境是怎麼做出來的、包含什麼、有什麼事要特別注意的？』

 

專案經理：『多一個人知道開發環境怎麼架設也好，也許下個專案你也可以幫忙建立開發環境，我會請系統組組長花點時間向你講解開發環境的各個環節。首先我先列出開發環境該有的項目：』

 

  1. 整合開發工具
  2. Cross compiler
  3. 制作執行檔的批次檔(build.bat)
  4. Makefile
  5. Link Script
  6. 除錯工具
  7. 下載工具
  8. 其他工具(offline tools)
  9. 模擬器
  10. 版本控制工具

 

5.1.     開發工具

 

系統老鳥：『經過剛剛的報告，你對開發環境的使用還有問題嗎？』

 

菜鳥：『報告長官，我前幾天就已經先用過了，基本上沒什麼問題，因為步驟真的相當簡單。只要從server上把所有的程式與工具下載到自己的電腦，只要路徑設定對，程式沒問題的話，在DOS視窗執行build.bat，它好像會去做很多事情，過一陣子就會自動把執行檔產生出來；然後再利用工具把執行檔下載到機器就可以了。對寫系統程式或應用程式的同仁相當簡單，完全不需理會細節。但是我對build.bat裡面做的事情相當有興趣，我希望我以後也能夠自己建立嵌入式系統的開發環境。』

 

系統老鳥：『PM老大有交代過，要我把其中的技術教你一下。首先我先問你，以你對嵌入式系統開發的了解，從大家把程式寫好之後，到要生成執行檔，其中必須包含哪些步驟？』

 

菜鳥：『之前PM老大曾讓我在板子上，寫了一個簡單且獨立的小程式，以下的動作在每次改完程式要測試之前都要做一次：』

 

  1. 編譯各個程式
  2. 編輯Link Script，指定程式要被定位到哪個位址
  3. 連結步驟a產生的所有目的檔(object file)，產生執行檔
  4. 用ICE下載並除錯
  5. 如果程式無誤則將其燒到ROM[1]中

 

系統老鳥：『這些是製作執行檔最基本的步驟；但你要知道，軟體專案並不是只有程式而已，往往還包含一些資源檔，例如圖檔、字型、聲音檔等，更複雜的專案可能還需要一個檔案系統。要如何把這所有的東西一一塞到ROM中就是一件麻煩的事情；此外專案內的程式檔可能相當多，無可避免的要寫一個或多個makefile，總不能要工程師一個一個程式自己打指令去編譯吧。再加上不同程式可能要在不同的位址執行、而且你要每一個應用程式工程師都要會用ICE除錯、用燒錄器燒ROM簡直有點為難人了。所以為了節省大部分研發人員時間，有許多大大小小的工作要作，雖然每一個電子產品專案所需的開發環境都大不相同，但基本上你剛列出來的項目還得再補上下列步驟：有的西要案的檔情』

 

  1. Makefile寫作
  2. Offline tools – 將不同的資源檔作成可加入ROM的二元檔(binary file)
  3. ROM-Maker – 將執行檔以及其他資源檔製作成可燒錄到ROM的二元檔
  4. Simulator
  5. Update Tool – 是一個PC程式，可透過某種方式(如USB、網路或RS232)與機器上的程式(稱為update-program)連接，讓工程師可以不用透過燒錄器就可以更新機器上的程式。

 

菜鳥：『我覺得這是個很具挑戰性的工作，因為一定會對整個軟體專案的細節最為了解，所以通常都會在專案的核心之內；我能感覺大PM非常信任你，才會把這個工作交給你。』

 

系統老鳥：『還好啦，開發環境是嵌入式系統開發不可或缺的項目，然而因為工作內容很雜，而且極度與個別產品相關，所以一般書籍都不會提到。我們就以目前這個開發案為例，盡量把基本觀念傳達給你。有的西要案的檔情』

IDE (整合開發環境)

菜鳥：『難道嵌入式系統就沒有像Visual Studio那樣的整合開發環境嗎(IDE – Integrated Developing Environment)？終究自己寫makefile與link script還是蠻麻煩的。』

 

系統老鳥：『其實許多比較通用的CPU所搭配的開發環境還是有IDE可以用，例如ARM、8051等，下圖就是一個商用的8051整合開發環境的畫面，功能相當齊全，有基本的軟體專案管理、程式編輯區、對應的組合語言、顯示目前CPU暫存器與記憶體的狀態以及debug功能等等；工程師只要用這個環境就可以完成開發、除錯、模擬、以及製作二元執行檔等所有動作，在比較簡單的開發專案，這種環境其實還蠻方便的。』

圖5-1：功能較完整的整合開發環境

 

菜鳥：『既然這樣，我們何不使用CPU廠商建議的IDE就好了，為什麼還要自己去弄一套開發環境出來？』

 

系統老鳥：『以我的經驗來說，即便是功能齊備的商業級IDE也很難完全符合每一個專案的應用需求，越複雜的專案更是如此；就以剛剛我們所談到的開發環境該具備的功能來說，商業級IDE就很難整合我們自己開發ROM-Maker以及offline tools。再以軟體專案管理來說，IDE最後也是產生makefile和link script，並據此製作執行檔；如果我們有比較特殊的應用，最終也是要手動去更改這些檔案。與其如此，不如一開始就依據專案的需求自行編輯相關檔案。

 

而且剛剛看的圖示是8051的例子，如果我們選用的不是那麼通用的CPU，恐怕就沒有功能那麼強的IDE可以用了；假如使用這些CPU的廠商並不是那麼多，根本不會有專業的開發環境廠商(如Metrowerks的CodeWarrior Development Tools)幫其開發IDE，所以通常都是由CPU廠商自行開發，自然功能會比較陽春、使用者介面比較不friendly等。尤其是寫程式時一定要用到的文字編輯器(editor)，對用慣UltraEdit或Visual Stdio的工程師而言，上述IDE所提供的editor簡直是難用的不得了。

 

實務上，通常我們會捨棄IDE不用；開發程式的工程師可以使用任何自己習慣的editor，然後使用專案團隊自行設計的開發環境，只要一個指令就可以做完所有的事情，最後產生可直接下載到ROM的執行檔。』

圖5-2：功能較不完整的整合開發環境

 

菜鳥：『難道CPU廠商提供的IDE真的如此不堪一用嗎？』

 

系統老鳥：『還不至於完全用不到，通常我會把它定義為遠端除錯(Remote debugger[2])工具；我們可以自己開發ROM-Maker以及下載工具，但是Remote Debugger就像Emulator一樣，是不可能自己開發的。而系統程式及應用程式的除錯主要還是在Simulator之上，只有特殊狀況才真的需要做遠端除錯(例如Simulator上OK，但機器上NG的bug)，所以除了驅動程式開發者之外，其他工程師要用到IDE的機會還真的不多。』

 

Cross-Tools

系統老鳥：『你認為在為新平台設計開發環境時，最先要考慮的是什麼？』

 

菜鳥：『是makefile和Link Script嗎？因為我覺得這兩個東西最麻煩。』

 

系統老鳥：『你會覺得麻煩一來是因為你不熟悉，二來因為makefile和link script的語法比較不直覺，而且很多人喜歡把它們寫得莫測高深；其實它們的觀念是很單純的。

 

話說回來，開始設計開發環境的時間點應該會在專案的早期，此時很多事情都還沒確定，很難掌握需要哪些步驟才可以做出執行檔；再加上不同CPU的平台通常會採用不同的開發工具，所以我認為在剛開始時最好花點時間去研究一下這些工具，看看到底提供了哪些功能，使用時有沒有特殊的注意事項，這在專案後期除錯或最佳化時都可能幫得上忙，我們最好心裡有個底；等系統設計漸趨完成時，再來開始寫makefile也不遲。在嵌入式系統開發中，我們把編譯器等工具稱為cross-tools，我不知道你是否清楚所謂“cross”的觀念？』

 

菜鳥：『這個之前韌體組的學長曾經教過我，所謂的“cross”的意義就是 - 要在PC上產生其他機器的執行檔。回想PC程式的產生流程為：

 

l   編譯(compile)

l   組譯(Assemble，有的編譯器只會產生組合語言碼，此時必須經過組譯才會產生目地碼 – Object file)

l   尋找標準函式庫

l   連結所有需要的object file與函式庫

l   產生執行檔(.EXE)

 

所以cross-tools應該包含：

l   Cross-Assembler

l   Cross-Compiler

l   Cross-Library

l   Cross-Linker

』

 

系統老鳥：『基本上觀念沒錯，但我們會用到的cross-tools應該還包含：

 

l   執行檔格式轉換工具：業界常用的執行檔格式很多，例如Windows上的EXE以及DLL，或業界中嵌入式系統開發環境用之有年的HEX與S-Record格式，以及最近常用的ELF[3]。

 

l   執行檔dump工具：將執行檔dump出(或“轉換為”)組合語言碼，這個工具在開發後期的效能分析與最佳化時相當有用。

 

l   Debug library與工具：例如知名的GNU gdb，只要系統中連結了debug library，就可以透過序列埠、串列埠、USB線或網路線，與PC端的debug程式連接，工程師就可以在PC上進行遠端除錯。

 

可使用的工具可能會隨平台不同而天差地別，所以在設計開發環境之前，負責的同仁一定要詳讀user manual，弄清楚這套工具的極限在哪裡。』

 

菜鳥：『如果要每一個應用程式工程師都要搞懂這些事情，的確是蠻令人頭痛的一件事；所以還是讓專人去處理會比較有效率。我還有一些問題，你剛剛提到要先注意cross-tools是否有提供某些特殊功能，我想知道所謂特殊的功能有哪些？』

 

系統老鳥：『主要的“特殊”功能都在編譯器上，其實也不是多特殊啦；我以已經被移植到多個平台的開發環境 – GNU Tool-Chain裡的C編譯器為例，以下是一些編譯時期的選項：

 

l   -Werror：將所有警告訊息變成錯誤訊息，亦即一旦有警告訊息產生，就不會產生object file。

l   -S：編譯時期順便輸出組合語言碼

l   -E：只執行C preprocessor，不產生object file

l   -D：編譯時期才定義巨集常數，效果如同在程式中使用 #define XXX

l   -O、-O2、-O3：最佳化等級，等級越高，產生的程式越有效率，相對的可能會比較不穩定

l   -g：編譯時加入debugging information，使GDB可以進行遠端除錯。

l   …

 

GNU Tool-Chain裡的Linker，則有以下參數可使用：

 

l   -T：指定link script，其中會指定要連結哪些檔案，並分別定址到哪個位址。

l   -Map：連結時產生map檔，map檔中包含了程式中所有symbol的位址資訊，在除錯時可以幫上很大的忙[4]。

 

此外，有些Linker還可以產生不同格式的執行檔；舉例來說，GNU tool chain可以支援許多不同的CPU，而且這套工具和公用函式庫都是開放原始碼，使用者可以根據其需求設定其組態(configuration)自行製作GNU tools；可以從其工具的名稱得知其基本屬性，例如arm-elf-gcc就是會以ELF格式產生ARM機器碼的C編譯器，而68K-coff-ld就是會以COFF格式產生68000機器碼的linker。』

 

菜鳥：『既然每一個專案使用的cross compiler都可能不一樣，我怎麼知道用什麼參數，產生什麼格式的執行檔才是對的？』

 

系統老鳥：『具體來說，你這個問題並沒有標準答案，隨專案需求而定。例如一般狀況，編譯時期是不需要產生組合語言碼的，只有在除錯或程式加速時，才有檢查程式組合語言碼的需求。但有個大原則是：剛開始時只要按照使用手冊或廠商sample code中的建議來設定參數，先讓系統可以運行無誤；此後再根據需求修改參數即可。』

 

Make

 

系統老鳥：『除了compiler、linker等工具外，開發環境中還有一個重要的東西 – make，我知道一般工程師都還蠻抗拒使用這個工具的，你以前有自己寫過makefile嗎？』

 

菜鳥：『我在Linux上寫程式時有寫過makefile，不過都是拿別人寫好的範例來改的，而且只會用很簡單的技巧而已；此外我在Linux上看過一些很複雜的makefile，幾乎完全看不懂它在寫什麼！』

 

系統老鳥：『有些cross-tools會提供圖形化的makefile與link script的編輯工具(如圖5-4)，如果系統中的source tree與重定位的需求不那麼複雜，使用這些工具並無不可。實際上makefile與link scrip的功能頗為強大，而上述工具往往缺乏彈性，碰到一些比較複雜的狀況只能束手無策。舉例來說，為了加速，系統中有幾個模組必須載入到CPU內部的RAM(internal RAM)裡執行，但internal RAM的size很小，無法同時容納這些模組；所以這些大小不一的模組必須定址到同一個位址(即internal RAM的起始位址)，執行時期有需要時再分別將其載入。這種狀況在嵌入式系統開發中並不算複雜，但一般makefile與link script的自動化編輯工具並無法處理這種狀況。

 

圖5-3：較複雜的重定位狀況，自動化link script編輯工具通常無法處理

 

我認為身為嵌入式系統的開發者，必須正視不同的開發案會有不同的開發環境、不同的硬體與記憶體配置、不同的source tree、不同的函式庫…等狀況，會碰到什麼複雜的應用根本無法想像，而且不見得每次都有這種自動化工具可用，所以建議你還是多學習一下makefile與link script的基本觀念和語法比較好，其實並不會太難。』

 

圖5-4：有些Cross-Tools會包含可自動產生makefile與Link-Script的工具

 

系統老鳥：『makefile勉強可以算是一種程式語言，它的語法不像C語言那麼直覺；其實什麼語言都一樣，有人可以把程式寫得人人都看得懂，也有人可以把程式寫得莫測高深，最後連自己都看不懂。對makefile我個人的看法是：它不像系統程式有效能、code size等考量，如果可以的話，我們只要用到makefile最基本的技巧就可以了。此外，makefile也可以寫註解，註解可以消除語法的不直覺；你跟在PM老大身邊也有一段時間了，你應該知道他最討厭那種不容易看得懂的程式。』

 

菜鳥：『嗯，我是看過他為了這種事情飆某個工程師，當初我是覺得哪有這麼嚴重，後來他分析給我聽，在他的管理的立場，控制程式的可讀性確實是很重要的事情。我還有一點搞不懂，make不是UNIX的東西嗎？嵌入式系統的cross-tools也會提供這個工具嗎？』

 

系統老鳥：『你可能對make的觀念有點偏差，第一，make這個工具基本上和哪一套開發工具無關，你可以用同一個make工具來建構不同平台的執行檔；簡單的說，make是用來進行自動編譯的程式工具，你只要在makefile中詳細敘述要用什麼工具(例如產生ARM機器碼的cross-compiler)，對哪一個程式(.c)或檔案(.obj…)做怎麼樣的處理(例如產生不要最佳化的object file)，make就會檢查這些檔案是否已“過期”[5]，如果有較新的版本，make就會依照makefile裡的規定去處理這個檔案。

 

第二，make這個工具不是UNIX的專利，PC上的開發環境也都會用到make，只是一般人在PC上都是使用整合開發環境來寫程式，所以不知道執行檔產生的細節。例如微軟開發環境的make工具叫nmake.exe，而Borland C++ builder的make工具就叫做make.exe。』

 

菜鳥：『你的意思是說make工具除了可以依據makefile的內容去執行compiler、linker等程式外，最主要的功能是可以判斷某個程式是否有新版本，而且只會處理這些有新版本的程式，它到底是怎麼做到的？』

 

 


[1] 嵌入式系統有各式各樣的硬體架構，但通常不會是PC的架構，基本上程式是直接放在如Mark-ROM或Flash等記憶體中執行；其中Mask-ROM是唯讀的，開發時期通常會用可多次寫入的EEPROM或NOR-Flash代替。本章中統一稱其為ROM，總之就是儲存並執行程式的記憶體，(詳情可參閱第5.4節)

[2] 遠端除錯 – 通常嵌入式CPU的IDE都會提供除錯的功能，實際上它是在PC上透過某種方式與機器的CPU連接，在PC上對遠端機器上執行的程式作除錯，所以又稱為遠端除錯。

[3] 執行檔格式 – 業界常用的執行檔格式很多，這些執行檔不是單純的二元檔，其中還包含了位址資訊以及除錯資訊。所以一般ICE、燒錄器或除錯程式接受的執行檔格式通常都不是單純的二元檔，而編譯工具產生的執行檔格式可能和上述工具產生的格式不同，所以需要許多轉檔程式。

[4] Map檔：有關map檔的應用，請參考第5.3節。

[5] 舉個簡單的例子，如果某個程式檔的最後更改日期比執行檔還要新，這表示上次產生執行檔之後，該程式檔已做了修改，則此時應該重新編譯、連結，以產生符合目前版本程式的執行檔。
